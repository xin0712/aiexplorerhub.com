:::tip 先看结论
企业级 Agent 架构的核心，不是让模型更聪明，而是让 Agent 在真实业务里安全、稳定、可控、可审计、可扩展、低成本地长期运行。

这篇你会掌握三件事：

- 企业级 Agent 为什么比普通 Agent 复杂得多
- 一套可落地的企业级 Agent 分层架构怎么设计
- 从 MVP 到企业级系统的演进路径与设计原则
:::

前面我们讲了单 Agent、Runtime、Planner、Controller、Harness，以及 Multi-Agent 多智能体系统。

这些概念偏向“Agent 如何运行”。

但如果我们真的要把 Agent 用到企业里，就不能只关心它能不能回答问题，而要关心：

它能不能安全、稳定、低成本、可审计、可扩展地运行在真实业务中。

这就是企业级 Agent 架构要解决的问题。

普通 Agent 解决的是“能不能完成任务”。

企业级 Agent 解决的是：

能不能长期运行？  
能不能接入公司系统？  
能不能控制权限？  
能不能追踪每一步？  
能不能保证结果可靠？  
能不能控制成本？  
出了问题能不能回溯？

所以，企业级 Agent 架构的核心，不是把模型接进来就完事，而是围绕模型搭建一整套可控的业务执行系统。

## 一、为什么企业级 Agent 更复杂？

个人使用 Agent 时，容错空间比较大。

比如你让 Agent 帮你写一段文案，它写错了，你改一下就行。

但企业场景不同。

如果 Agent 接入了真实业务系统，它可能会：

查询客户数据  
生成投放策略  
修改账户预算  
调用内部接口  
访问数据库  
发送客户邮件  
生成财务报表  
处理合同信息  
执行自动化流程

这时，Agent 的每一次动作都可能影响真实业务。

所以企业级 Agent 必须重点考虑：

安全  
权限  
稳定性  
成本  
审计  
合规  
人工兜底

一句话：

企业级 Agent 不是一个聊天机器人，而是一个可以接入业务系统的智能执行系统。

## 二、企业级 Agent 的整体架构

一个企业级 Agent 系统，通常可以拆成以下几层：

用户入口层  
↓  
任务理解层  
↓  
规划与调度层  
↓  
工具与能力层  
↓  
权限与安全层  
↓  
数据与记忆层  
↓  
执行 Runtime 层  
↓  
观测、审计与评测层

可以画成这样：

```text
┌────────────────────────────────────┐
│            用户入口层               │
│ Web / 飞书 / 企业微信 / Slack / API │
└────────────────────────────────────┘
                 ↓
┌────────────────────────────────────┐
│            任务理解层               │
│ 意图识别 / 参数抽取 / 任务分类       │
└────────────────────────────────────┘
                 ↓
┌────────────────────────────────────┐
│          Planner / Controller       │
│ 任务拆解 / 流程调度 / 状态控制       │
└────────────────────────────────────┘
                 ↓
┌────────────────────────────────────┐
│           Tool / Skill 层           │
│ SQL / API / RPA / 搜索 / 文档 / 报表 │
└────────────────────────────────────┘
                 ↓
┌────────────────────────────────────┐
│        Runtime / 执行环境层          │
│ Python / Shell / Browser / DB / MCP │
└────────────────────────────────────┘
                 ↓
┌────────────────────────────────────┐
│       安全、权限、审计、评测层        │
│ Permission / Logs / Eval / Monitor  │
└────────────────────────────────────┘
```

这套架构的重点是：

<span style={{color: 'var(--ifm-color-primary)'}}><strong>模型只是中间的大脑，企业级架构真正重要的是外层控制系统。</strong></span>

## 三、核心模块一：用户入口层

用户入口层负责接收任务。

企业里的 Agent 不一定只存在于聊天窗口里，它可能出现在很多入口：

飞书机器人  
企业微信机器人  
Web 后台  
浏览器插件  
内部运营系统  
数据平台  
CRM  
BI 系统  
API 接口

例如广告投放 Agent，可以让用户在飞书里输入：

分析昨天 Nutra 业务 ROI 下滑原因

也可以在内部系统里点击一个按钮：

生成昨日投放复盘

还可以通过定时任务自动触发：

每天上午 10 点自动生成日报

所以企业级 Agent 的第一层，不是模型，而是业务入口。

入口层需要解决三个问题：

谁在使用？  
从哪里发起？  
任务上下文是什么？

例如同一句话：

帮我分析昨天数据。

不同人说，含义可能完全不同。

运营说，可能是分析投放数据。  
财务说，可能是分析收入数据。  
产品说，可能是分析功能使用数据。

<span style={{color: 'var(--ifm-color-primary)'}}><strong>所以入口层要尽可能带上用户身份、业务角色、入口来源和上下文。</strong></span>

## 四、核心模块二：任务理解层

任务理解层负责把用户的自然语言，转成系统能处理的任务结构。

用户说的是：

帮我看看昨天 SOI 为什么掉了

但系统真正需要理解的是：

任务类型：投放异常分析  
业务类型：SOI  
时间范围：昨天  
核心指标：ROI / revenue / cost / conversion  
输出形式：原因分析 + 优化建议  
是否需要执行动作：否

这一步通常包括：

意图识别  
实体抽取  
参数补全  
任务分类  
风险识别

比如：

用户输入：把 ROI 低于 0.8 的广告组都暂停掉  
任务类型：投放操作  
涉及动作：暂停广告组  
风险等级：高  
需要权限：是  
需要人工确认：是

企业级 Agent 不能听到一句话就直接执行。

它必须先判断：

这是查询任务？  
分析任务？  
生成任务？  
还是执行任务？

<span style={{color: 'var(--ifm-color-primary)'}}><strong>不同任务的安全等级完全不同。</strong></span>

## 五、核心模块三：Planner 规划层

Planner 负责把任务拆成步骤。

比如用户说：

分析昨天 SOI 业务 ROI 下滑原因。

Planner 可能会拆成：

1. 查询昨天 SOI 的整体消耗、收入、ROI
2. 对比前一天或近 7 日均值
3. 分析 CTR、CVR、purchase、event10 是否异常
4. 按 channel / bundle / tagid / adv_id 拆分
5. 找出异常贡献最大的维度
6. 判断是流量质量问题、出价问题、转化问题还是数据回传问题
7. 输出原因和建议

Planner 的价值是让 Agent 不只是“凭感觉回答”，而是按照明确的业务链路分析。

在企业场景中，Planner 最好不要完全自由发挥。

更好的方式是：

<span style={{color: 'var(--ifm-color-primary)'}}><strong>让 Planner 基于预定义业务 SOP 进行规划。</strong></span>

例如投放异常分析可以有固定 SOP：

先看大盘  
再看行业  
再看广告主  
再看渠道  
再看素材  
再看转化漏斗  
再看作弊率  
最后输出建议

这样既有智能性，又有稳定性。

## 六、核心模块四：Controller 调度层

Controller 负责控制执行过程。

Planner 说“要查数据”，但具体怎么查、什么时候查、查失败怎么办，是 Controller 的工作。

Controller 需要处理：

当前执行到哪一步  
下一步调用哪个工具  
工具返回是否正常  
是否需要重试  
是否需要换一种方式  
是否需要重新规划  
是否需要人工确认  
什么时候停止

举个例子：

Planner：查询昨天 SOI 的 ROI 数据  
Controller：调用 SQL Tool  
SQL Tool：报错，表不存在  
Controller：尝试备用表  
备用表：返回数据为空  
Controller：检查日期参数  
发现昨天数据还未产出  
Controller：返回“数据未就绪”，而不是继续胡说

这就是企业级 Agent 和普通聊天机器人的区别。

普通聊天机器人可能会根据经验猜一个答案。

<span style={{color: 'var(--ifm-color-primary)'}}><strong>企业级 Agent 必须尊重真实数据和执行状态。</strong></span>

## 七、核心模块五：Tool / Skill 能力层

企业级 Agent 必须能调用业务工具。

常见工具包括：

数据库查询工具  
BI 查询工具  
搜索工具  
文档读取工具  
邮件工具  
飞书通知工具  
CRM 工具  
广告平台 API  
财务系统 API  
工单系统  
代码执行工具  
RPA 浏览器工具

但这些工具不能随便开放。

每个工具都需要定义清楚：

工具名称  
工具用途  
输入参数  
输出格式  
权限要求  
风险等级  
是否需要人工确认  
调用频率限制  
失败处理方式

例如广告平台操作工具：

工具：pause_ad_group  
作用：暂停广告组  
参数：ad_group_id  
风险等级：高  
权限要求：投放管理员  
人工确认：必须  
审计记录：必须

再比如 SQL 查询工具：

工具：query_ads_data  
作用：查询投放数据  
参数：date、adv_id、channel、metric  
风险等级：中  
权限要求：数据查看权限  
人工确认：不需要  
审计记录：需要

<span style={{color: 'var(--ifm-color-primary)'}}><strong>Tool 层的核心原则是：</strong></span>

<span style={{color: 'var(--ifm-color-primary)'}}><strong>能查和能改必须分开，低风险和高风险必须分级。</strong></span>

## 八、核心模块六：Runtime 执行环境层

Runtime 是 Agent 真正执行动作的地方。

例如：

运行 SQL  
执行 Python  
调用 API  
打开网页  
读取文件  
生成报表  
发送消息

企业级 Runtime 不能是完全开放的。

它必须具备隔离能力。

例如：

不能随便访问所有数据库  
不能随便执行系统命令  
不能访问未授权文件  
不能无限制调用外部 API  
不能把敏感数据发到外部

所以企业级 Runtime 通常需要：

沙箱隔离  
资源限制  
网络限制  
权限控制  
调用白名单  
超时机制  
失败回滚  
日志记录

比如：

SQL Runtime 只能查询授权表  
Python Runtime 不能访问生产密钥  
API Runtime 只能调用白名单接口  
Browser Runtime 不能自动提交高风险表单

一句话：

<span style={{color: 'var(--ifm-color-primary)'}}><strong>Runtime 不是越自由越好，而是越可控越好。</strong></span>

## 九、核心模块七：权限隔离

<span style={{color: 'var(--ifm-color-primary)'}}><strong>权限隔离是企业级 Agent 最关键的部分之一。</strong></span>

因为 Agent 一旦接入公司系统，就可能接触：

客户数据  
账户数据  
投放预算  
合同信息  
财务数据  
员工信息  
商业机密

所以必须回答三个问题：

这个用户是谁？  
他能看什么？  
他能操作什么？  
这个 Agent 能代表他做什么？

企业级 Agent 不能只有一个万能账号。

更合理的方式是：

Agent 继承用户权限，而不是拥有超级权限。

例如：

运营人员只能查看自己负责的客户；  
财务人员只能查看财务相关数据；  
产品经理可以看汇总数据，但不能操作投放账户；  
投手可以调整广告组，但需要在自己负责范围内。

权限可以按几个维度设计：

用户角色  
业务线  
客户  
广告主  
数据表  
字段级权限  
工具级权限  
操作级权限

例如：

用户 A 可以查询 Nutra 数据，但不能查询保险业务数据  
用户 B 可以看 ROI，但不能看客户回款明细  
用户 C 可以生成暂停建议，但不能真正暂停广告组

这就是权限隔离。

## 十、核心模块八：人工确认机制

<span style={{color: 'var(--ifm-color-primary)'}}><strong>企业级 Agent 不能所有动作都自动执行。</strong></span>

<span style={{color: 'var(--ifm-color-primary)'}}><strong>尤其是涉及真实业务影响的操作，必须有人确认。</strong></span>

可以把 Agent 的动作分成三类：

| 动作类型 | 示例 | 是否需要确认 |
| --- | --- | --- |
| 低风险 | 查询数据、生成报告、总结文档 | 一般不需要 |
| 中风险 | 发送内部通知、创建草稿、生成配置建议 | 可按规则确认 |
| 高风险 | 改预算、暂停广告组、发客户邮件、删除数据 | 必须确认 |

例如用户说：

把 ROI 低于 0.8 的广告组都暂停

Agent 不应该直接执行。

正确流程应该是：

1. 查询 ROI 低于 0.8 的广告组
2. 生成待暂停列表
3. 标明影响范围
4. 给出原因
5. 请求用户确认
6. 用户确认后再执行
7. 执行后输出结果和审计记录

这就是 Human-in-the-loop，人在环。

企业级 Agent 的核心不是完全无人化，而是：

让 AI 做分析和建议，让人控制关键决策。

## 十一、核心模块九：审计与日志

<span style={{color: 'var(--ifm-color-primary)'}}><strong>企业级 Agent 的每一步都必须可追踪。</strong></span>

因为一旦出了问题，需要知道：

是谁发起的？  
什么时候发起的？  
Agent 理解成了什么任务？  
调用了哪些工具？  
读取了哪些数据？  
生成了什么中间结果？  
执行了什么动作？  
谁确认的？  
最终结果是什么？

审计日志至少应该记录：

用户输入  
任务解析结果  
计划步骤  
工具调用参数  
工具返回结果  
模型输出  
人工确认记录  
最终动作  
异常信息  
时间戳  
操作者身份

例如：

2026-05-28 10:05 用户提交：分析昨天 SOI ROI 下滑原因  
2026-05-28 10:06 Agent 调用 query_roi_data  
2026-05-28 10:06 返回 ROI = 0.72  
2026-05-28 10:07 Agent 调用 query_channel_breakdown  
2026-05-28 10:08 发现 vungle 渠道 CVR 下滑 35%  
2026-05-28 10:09 Agent 生成建议：降低 vungle 出价 15%  
2026-05-28 10:10 用户确认：仅生成建议，不执行

审计的价值是：

可回溯  
可复盘  
可追责  
可优化  
可合规

没有审计的 Agent，不适合进入企业核心业务。

## 十二、核心模块十：成本控制

企业级 Agent 的成本不只是模型费用。

它还包括：

模型调用成本  
工具调用成本  
数据库查询成本  
API 请求成本  
计算资源成本  
人工审核成本  
错误执行成本

Agent 越智能，越容易产生高成本。

例如一个任务可能会：

多次调用大模型  
多次查询数据库  
反复重试  
读取大量文档  
调用多个专家 Agent  
生成很长的中间结果

如果没有成本控制，系统很快会失控。

常见成本控制方式包括：

模型分层  
缓存机制  
查询限额  
任务限流  
结果复用  
小模型优先  
大模型兜底  
限制最大循环次数  
限制上下文长度  
限制工具调用次数

比如：

简单分类任务用小模型  
复杂推理任务用大模型  
固定规则判断不用模型  
重复查询结果直接走缓存  
低价值任务限制最大 token  
高成本任务需要用户确认

<span style={{color: 'var(--ifm-color-primary)'}}><strong>企业级 Agent 的设计原则是：</strong></span>

<span style={{color: 'var(--ifm-color-primary)'}}><strong>不是所有问题都要用最强模型解决。</strong></span>

## 十三、核心模块十一：稳定性与容错

<span style={{color: 'var(--ifm-color-primary)'}}><strong>企业环境里，系统不能因为一次工具失败就崩掉。</strong></span>

Agent 要能处理各种异常：

数据库超时  
接口失败  
权限不足  
参数缺失  
数据为空  
模型输出不合规  
工具返回格式错误  
任务执行超时  
网络异常

所以需要容错机制：

重试  
降级  
备用工具  
超时终止  
回滚  
人工接管  
错误解释  
状态恢复

例如：

主数据表查询失败 -> 查询备用聚合表  
实时数据未产出 -> 使用昨日离线数据并提示口径差异  
模型生成失败 -> 使用模板生成基础报告  
自动执行失败 -> 转为人工工单

企业级 Agent 必须避免两种危险情况：

失败后继续胡编  
失败后无声中断

正确做法是明确告诉用户：

哪一步失败了  
为什么失败  
影响范围是什么  
当前能给出什么结果  
需要用户做什么

## 十四、核心模块十二：评测与质量控制

<span style={{color: 'var(--ifm-color-primary)'}}><strong>企业级 Agent 不能只靠“看起来不错”。</strong></span>

<span style={{color: 'var(--ifm-color-primary)'}}><strong>必须有评测体系。</strong></span>

常见评测指标包括：

任务完成率  
工具调用成功率  
答案准确率  
幻觉率  
人工接管率  
平均执行时间  
平均成本  
用户满意度  
高风险动作拦截率  
错误恢复成功率

不同任务有不同评测方式。

例如数据分析 Agent，需要评估：

SQL 是否正确  
指标口径是否正确  
异常原因是否命中  
建议是否可执行  
是否遗漏关键维度

客服 Agent 需要评估：

回答是否符合知识库  
是否泄露隐私  
是否错误承诺  
是否解决问题

投放 Agent 需要评估：

建议是否提升 ROI  
是否控制消耗风险  
是否误伤优质广告组  
是否能解释原因

企业级 Agent 的优化，不是一次性上线，而是持续迭代：

上线  
收集日志  
人工评审  
发现错误模式  
优化 Prompt / Tool / SOP  
重新评测  
继续上线

## 十五、企业级 Agent 的典型落地架构

以“广告投放 Agent”为例，可以设计成这样：

飞书 / Web 后台  
↓  
任务入口  
↓  
任务理解层  
↓  
Supervisor Agent  
↓  
数据分析 Agent  
流量质量 Agent  
投放策略 Agent  
素材分析 Agent  
合规审核 Agent  
↓  
SQL Runtime / API Runtime / Python Runtime  
↓  
权限系统 / 审计日志 / 成本监控  
↓  
生成报告 / 给出建议 / 等待人工确认  
↓  
必要时执行投放动作

一个完整任务可能是：

用户：分析昨天保险业务为什么 ROI 下滑，并给出优化建议。

系统执行过程：

1. 判断任务类型：投放异常分析
2. 识别业务类型：保险
3. 识别时间范围：昨天
4. 调用数据分析 Agent
5. 查询消耗、收入、ROI、CTR、CVR、purchase
6. 调用流量质量 Agent
7. 分析 affiliate、bundle、tagid、作弊率
8. 调用策略 Agent
9. 判断出价、预算、渠道结构是否异常
10. 调用合规审核 Agent
11. 检查输出是否可对外发送
12. 生成最终报告
13. 如涉及调价或暂停，进入人工确认

最终输出不是简单一句：

ROI 下降，建议优化。

而是结构化结论：

核心原因：  
昨日保险业务 ROI 下滑主要由 vungle 渠道 CVR 下降导致。  
数据表现：  
消耗上升 18%，收入下降 12%，ROI 从 1.05 降至 0.76。  
其中 vungle 渠道消耗占比提升至 42%，但 purchase 下降 31%。  
可能原因：  
1. 新增流量包转化质量偏弱  
2. 部分 bundle 点击率高但后端转化低  
3. 出价提升后进入更宽流量池，导致质量波动  
建议动作：  
1. 降低低 ROI bundle 出价 10%-15%  
2. 暂停连续 2 天无 purchase 的 tagid  
3. 保留 ROI > 1 的优质渠道继续扩量  
4. 明日重点监控 CVR、purchase、click_cheat_ratio

这才是企业级 Agent 应该产出的结果。

## 十六、企业级 Agent 的设计原则

最后，总结几个非常重要的设计原则。

### 1. 先做辅助决策，再做自动执行

不要一开始就让 Agent 自动改系统。

更稳的路径是：

先生成分析  
再生成建议  
再生成执行清单  
再人工确认  
最后逐步自动化

也就是：

Copilot -> Semi-Auto -> Auto

先做副驾驶，再做半自动，最后才考虑自动驾驶。

### 2. 让 Agent 使用 SOP，而不是自由发挥

企业场景里，稳定性比创意更重要。

因此应该把业务经验沉淀成：

分析模板  
诊断流程  
执行规则  
指标口径  
异常判断标准  
风险等级

Agent 不是完全自由思考，而是在 SOP 框架下进行判断。

### 3. 高风险动作必须可控

凡是涉及真实业务影响的动作，都要有：

权限校验  
影响预估  
人工确认  
操作日志  
回滚方案

例如：

暂停广告组  
修改预算  
删除数据  
发送客户邮件  
提交合同  
调整价格

这些不能让 Agent 直接裸奔执行。

### 4. 工具能力要白名单化

不要给 Agent 一个万能接口。

而是给它明确、有限、可控的工具。

例如不要给它：

任意执行 SQL

而是给它：

query_roi_report  
query_channel_breakdown  
query_fraud_metrics  
query_postback_funnel

工具越结构化，Agent 越稳定。

### 5. 所有结果都要可追溯

企业级 Agent 的每个结论，最好都能追到来源：

数据来自哪里  
计算口径是什么  
判断规则是什么  
调用了哪个工具  
中间结果是什么  
最终建议怎么得出

否则用户很难信任它。

## 十七、从 MVP 到企业级 Agent 的演进路径

如果你要从 0 开始做企业级 Agent，不建议一上来就做全自动。

可以分四个阶段。

### 阶段一：问答型 Agent

先让它能回答业务问题：

昨天 ROI 是多少？  
哪个渠道消耗最高？  
哪个广告组转化最差？

目标是打通数据查询和基础问答。

### 阶段二：分析型 Agent

让它能做结构化分析：

为什么 ROI 下滑？  
哪个维度贡献最大？  
CTR 和 CVR 哪个问题更大？

目标是形成分析链路。

### 阶段三：建议型 Agent

让它能给出可执行建议：

哪些广告组应该降价？  
哪些 bundle 应该暂停？  
哪些渠道可以扩量？

目标是辅助业务决策。

### 阶段四：半自动执行 Agent

让它在人工确认后执行动作：

生成待操作清单  
用户确认  
Agent 调用 API 执行  
输出执行结果  
记录审计日志

目标是提效，但不失控。

这条路径比直接做“全自动 Agent”安全得多。

## 总结

企业级 Agent 架构的核心，不是让模型变得更聪明，而是让 Agent 在真实业务中变得：

安全  
稳定  
可控  
可审计  
可扩展  
低成本  
可持续优化

普通 Agent 关注的是：

能不能完成任务。

企业级 Agent 关注的是：

能不能在真实企业环境中长期可靠地完成任务。

它需要的不只是 LLM，还需要：

权限系统  
工具系统  
Runtime  
Controller  
Planner  
Harness  
审计日志  
人工确认  
评测体系  
成本控制  
异常处理

可以用一句话记住：

企业级 Agent 不是一个会说话的模型，而是一套接入企业业务系统的智能执行架构。
