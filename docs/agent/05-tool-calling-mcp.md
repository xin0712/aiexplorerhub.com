:::tip 先看结论
Tool Calling 让 Agent 从“只会说”变成“可以做”，MCP 让 Agent 以统一方式连接越来越多的工具与数据源。

这篇你会理解三件事：

- Tool Calling 的完整执行链路，以及为什么参数和错误处理决定可用性
- MCP 解决的核心问题：工具标准化、上下文标准化、生态可扩展
- Tool Calling 与 MCP 的关系，以及它们如何共同构成可落地的 Agent 执行能力
:::

上一篇我们讲了 Memory，解决的是 Agent 的“记忆问题”。

但光有记忆还不够。

如果一个 Agent 只能在脑子里思考，却不能调用外部工具，那它本质上还是一个“会聊天的模型”。它可以分析、推理、规划，但不能真正替你完成事情。

比如你问它：

帮我查一下今天 SOXL 的走势，然后结合我的持仓，判断要不要减仓。

如果它没有工具能力，它只能基于已有知识泛泛分析。

但如果它能调用工具，它就可以：

1. 查询实时行情；
2. 获取你的持仓数据；
3. 读取新闻和财报信息；
4. 调用策略模型计算风险；
5. 最后给出一个更接近真实交易场景的判断。

这就是 Tool Calling 的意义。

简单说：

Tool Calling 让 Agent 从“只会说”，变成“可以做”。

而 MCP，则进一步解决了一个更大的问题：

工具越来越多，Agent 怎么用统一的方式去连接、理解和调用这些工具？

## 一、什么是 Tool Calling？

Tool Calling，也叫 Function Calling，指的是大模型在对话过程中，根据用户的意图，自动判断是否需要调用某个外部函数、API 或工具。

举个最简单的例子。

用户问：

北京今天多少度？

模型自己并不知道实时天气，所以它需要调用天气查询工具。

工具可能长这样：

```json
{
  "name": "get_weather",
  "description": "查询某个城市的实时天气",
  "parameters": {
    "city": "string"
  }
}
```

当用户问“北京今天多少度”时，模型会判断：

我要调用 `get_weather` 这个工具，并且参数是：

```json
{
  "city": "北京"
}
```

然后外部系统执行这个工具，把天气结果返回给模型。

模型再把结果整理成自然语言回答用户：

北京今天气温 18-26℃，天气晴，适合外出。

这里面其实分成三步：

第一步，模型理解用户意图。  
第二步，模型选择合适的工具，并生成参数。  
第三步，工具执行完成后，模型基于结果继续回答。

所以 Tool Calling 的核心不是模型自己去执行代码，而是：

模型负责判断“该调用什么工具、传什么参数”，真正执行的是外部系统。

## 二、为什么 Agent 必须有 Tool Calling？

因为大模型本身有几个天然限制。

第一，它不知道实时信息。

比如股价、天气、新闻、库存、订单状态，这些都是不断变化的。

第二，它不能直接操作外部系统。

比如发邮件、查数据库、建日程、下单、调接口，这些都不是纯语言生成能完成的。

第三，它不能访问你的私有数据。

比如你的 CRM、Notion、飞书文档、广告后台、数据库、代码仓库，这些都需要权限和接口。

所以，如果没有 Tool Calling，Agent 只能停留在“建议层”。

有了 Tool Calling，Agent 才能进入“执行层”。

比如：

| 用户需求 | 没有工具的模型 | 有工具的 Agent |
| --- | --- | --- |
| 查天气 | 只能猜或说无法获取实时信息 | 调用天气 API |
| 分析股票 | 只能讲通用逻辑 | 调用行情、新闻、财报数据 |
| 写日报 | 只能生成模板 | 读取数据表后生成日报 |
| 订会议 | 只能告诉你怎么做 | 调用日历工具创建会议 |
| 查广告效果 | 只能解释指标 | 查询数据库并输出分析 |

所以 Agent 的能力边界，很大程度取决于它能调用哪些工具。

## 三、Tool Calling 的基本流程

一个完整的 Tool Calling 流程通常是这样的：

用户提出需求  
↓  
模型理解用户意图  
↓  
判断是否需要调用工具  
↓  
选择合适工具  
↓  
生成工具参数  
↓  
外部系统执行工具  
↓  
返回执行结果  
↓  
模型基于结果继续推理  
↓  
输出最终答案

举个更具体的例子。

用户说：

帮我查一下 847 这个 adv_id 昨天的转化数和收入。

模型需要判断：

这不是一个普通聊天问题，而是一个数据查询任务。

它可能要调用 SQL 查询工具。

生成的 SQL 类似：

```sql
SELECT
  adv_id,
  SUM(conversion) AS conversion,
  SUM(revenue) AS revenue
FROM ad_report
WHERE adv_id = 847
  AND dt = CURRENT_DATE - INTERVAL '1' DAY
GROUP BY adv_id;
```

然后数据库返回：

```json
{
  "adv_id": 847,
  "conversion": 120,
  "revenue": 660
}
```

模型再回答：

昨天 adv_id = 847 的转化数是 120，收入是 660。如果按照单转化 5.5 计算，收入也正好是 120 × 5.5 = 660。

这里的关键点是：

模型不是直接“编答案”，而是通过工具拿到真实数据后再回答。

这也是 Agent 相比普通 Chatbot 更强的地方。

## 四、工具参数为什么很重要？

Tool Calling 看起来简单，但真正落地时，最容易出问题的是参数。

比如一个查天气工具需要的参数是：

```json
{
  "city": "string",
  "date": "string"
}
```

用户问：

明天上海会下雨吗？

模型就需要把自然语言转换成结构化参数：

```json
{
  "city": "上海",
  "date": "2026-05-28"
}
```

如果参数错了，工具调用就会失败。

常见错误包括：

1. 参数缺失；
2. 参数类型错误；
3. 参数格式错误；
4. 工具选择错误；
5. 用户表达模糊，模型误判。

比如：

用户说：

帮我查一下昨天的数据。

这里“昨天”是相对时间，需要根据当前日期转换成具体日期。

用户说：

查一下 847 的收入。

这里还缺少时间范围，模型需要追问：

你要查哪一天或者哪段时间的数据？

所以一个稳定的 Tool Calling 系统，必须做好三件事：

第一，工具描述要清晰。  
第二，参数约束要明确。  
第三，异常处理要完善。

## 五、工具调用失败时怎么办？

真实系统里，工具调用一定会失败。

比如：

数据库超时。  
API 限流。  
参数不合法。  
权限不足。  
接口返回空数据。

这时候 Agent 不能直接崩掉，也不能瞎编结果。

它应该能识别失败原因，并给出可执行的下一步。

比如数据库报错：

`Exceed concurrency limit`

Agent 应该解释：

当前 SQL 执行失败是因为数据库并发达到上限，不是 SQL 语法错误。可以稍后重试，或者缩小查询范围，减少扫描量。

再比如接口返回：

```json
{
  "error": "missing required parameter: date"
}
```

Agent 应该知道是缺少日期参数，然后向用户追问：

需要查询哪一天的数据？

这就是工具调用里的错误处理。

一个好的 Agent，不只是会调用工具，还要能理解工具失败后的结果。

## 六、Tool Calling 的局限

Tool Calling 很强，但也有几个明显问题。

第一个问题是，每个工具的接入方式都不一样。

天气 API 有自己的参数格式。  
数据库有自己的 SQL 规范。  
飞书、Notion、GitHub、Slack、Google Drive，每个平台都有自己的认证方式、接口规范和数据结构。

如果每接一个工具都要重新开发一套适配逻辑，成本会很高。

第二个问题是，工具描述不统一。

有些工具叫 `search_user`，有些叫 `query_customer`，有些叫 `get_profile`，本质上可能都是查用户，但模型很难自动理解它们之间的关系。

第三个问题是，工具权限不好管理。

一个 Agent 能不能读文件？  
能不能写数据库？  
能不能发邮件？  
能不能删除数据？

这些都需要统一的权限控制。

第四个问题是，工具越来越多以后，Agent 很难管理上下文。

如果一次性把几百个工具都塞给模型，模型会更容易选错工具，也会消耗大量上下文。

所以，Tool Calling 解决了“模型如何调用工具”的问题，但没有完全解决“工具如何标准化接入和管理”的问题。

这就是 MCP 出现的原因。

## 七、什么是 MCP？

MCP，全称是 Model Context Protocol，中文可以理解为“模型上下文协议”。

它的核心作用是：

用一种标准化的方式，把外部工具、数据源和上下文暴露给模型使用。

你可以把 MCP 理解成 Agent 世界里的“USB-C 接口”。

以前，每个工具都有自己的接口。

接 GitHub 要写 GitHub 的适配。  
接 Notion 要写 Notion 的适配。  
接数据库要写数据库的适配。  
接本地文件还要写本地文件的适配。

这就像以前各种设备都有自己的充电口，非常混乱。

而 MCP 想做的事情是：

让不同工具都按照统一协议接入，让 Agent 用统一方式发现工具、理解工具、调用工具。

所以 MCP 不是一个具体工具，而是一套连接标准。

## 八、MCP 解决了什么问题？

MCP 主要解决三个问题。

### 1. 工具标准化

以前工具接入是这样的：

Agent -> 自定义 API 1  
Agent -> 自定义 API 2  
Agent -> 自定义 API 3  
Agent -> 自定义 API 4

每个 API 都要单独适配。

有了 MCP 之后，可以变成：

Agent -> MCP Client -> MCP Server -> 外部工具/数据源

Agent 不需要关心每个工具底层怎么实现，只需要通过 MCP 协议访问。

### 2. 上下文标准化

Agent 不只是需要工具，也需要上下文。

比如：

代码文件。  
数据库表结构。  
文档内容。  
用户配置。  
项目说明。  
历史任务记录。

这些上下文如果没有标准方式提供给模型，模型就很难稳定理解当前环境。

MCP 可以把这些资源以标准格式暴露给模型。

### 3. 生态可扩展

当工具都按照同一套协议接入后，开发者可以写各种 MCP Server。

比如：

GitHub MCP Server。  
Postgres MCP Server。  
Google Drive MCP Server。  
Slack MCP Server。  
Browser MCP Server。  
File System MCP Server。

这样一来，Agent 的能力就可以像插件一样扩展。

你需要什么能力，就接入对应的 MCP Server。

## 九、MCP 和 Tool Calling 的关系

很多人容易把 MCP 和 Tool Calling 混在一起。

它们不是同一个东西。

Tool Calling 更像是一种“调用动作”。  
MCP 更像是一种“连接协议”。

可以这样理解：

| 概念 | 解决的问题 | 类比 |
| --- | --- | --- |
| Tool Calling | 模型如何调用工具 | 打电话 |
| MCP | 工具如何标准化接入模型 | 电话网络协议 |
| Agent | 负责思考、规划和执行 | 会安排任务的人 |

没有 MCP，也可以做 Tool Calling。

比如你自己写一个天气 API，然后让模型调用它。

但当工具数量变多，系统变复杂后，如果没有统一标准，维护成本会越来越高。

MCP 的价值在于：

让 Tool Calling 从“单点工具调用”，升级成“可扩展的工具生态”。

## 十、举个广告投放 Agent 的例子

假设我们要做一个广告投放 Agent。

它需要完成这些任务：

1. 查询广告消耗；
2. 分析 ROI；
3. 找出低转化 bundle；
4. 查询作弊率；
5. 生成优化建议；
6. 给运营发送日报。

如果不用 MCP，我们可能要分别接：

数据库 API  
广告后台 API  
反作弊系统 API  
飞书机器人 API  
报表系统 API

每一个都要定制开发。

但如果用 MCP，可以设计成：

广告数据 MCP Server  
反作弊 MCP Server  
飞书 MCP Server  
报表 MCP Server

Agent 通过统一协议发现这些工具：

可用工具：

- `query_ad_data`：查询广告数据
- `query_fraud_data`：查询反作弊数据
- `generate_report`：生成报表
- `send_feishu_message`：发送飞书消息

然后用户说：

帮我分析昨天 Nutra 的投放效果，找出 ROI 低于 0.8 的渠道，并发到飞书群。

Agent 的执行链路可能是：

1. 调用 `query_ad_data` 查询 Nutra 昨日数据
2. 按渠道聚合 spend、revenue、ROI
3. 筛选 ROI < 0.8 的渠道
4. 调用 `query_fraud_data` 查看是否作弊率异常
5. 生成优化建议
6. 调用 `send_feishu_message` 发送到群里

这才是一个真正可执行的 Agent。

它不是只告诉你“应该分析 ROI”，而是直接帮你把数据查出来、分析完、发出去。

## 十一、MCP 的真正价值

MCP 的价值不只是“接工具”。

它更大的价值是让 Agent 具备可扩展性。

过去我们做自动化系统，通常是流程固定的：

输入 -> 固定规则 -> 固定输出

比如每天定时跑 SQL，然后发送日报。

但 Agent 的特点是，它可以根据任务目标动态规划路径。

比如同样是“分析广告效果”，它可能根据不同情况选择不同工具：

ROI 低，就查渠道。  
CTR 低，就查素材。  
CVR 低，就查落地页。  
作弊率高，就查反作弊系统。  
预算没花出去，就查 bid、winrate、floor。

这种动态决策能力，需要大量工具支持。

而 MCP 就是为了让这些工具可以更容易地接进来。

所以 MCP 的核心价值是：

降低 Agent 连接外部世界的成本，让 Agent 更容易从“聊天助手”变成“任务执行系统”。

## 十二、总结

Tool Calling 解决的是：

模型如何调用工具。

MCP 解决的是：

工具如何标准化地接入模型。

Agent 真正要落地，必须同时具备三类能力：

第一，能理解任务。  
第二，能调用工具。  
第三，能基于工具结果继续推理和执行。

如果说大模型是 Agent 的大脑，Memory 是 Agent 的记忆，那么 Tool Calling 就是 Agent 的手脚。

而 MCP，则是连接这些手脚的标准接口。

没有 Tool Calling，Agent 只能说，不能做。

没有 MCP，Agent 可以做一些事，但工具接入会越来越混乱。

所以在 Agent 系统里，Tool Calling 和 MCP 是非常关键的一环。

它们共同决定了 Agent 能不能真正连接外部世界，完成真实任务。
