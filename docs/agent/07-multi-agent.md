:::tip 先看结论
Multi-Agent 的本质不是让更多模型同时输出，而是把复杂任务拆给不同角色协作完成，从而提升稳定性、可解释性和可扩展性。

这篇你会掌握三件事：

- 为什么复杂任务不能长期依赖单 Agent
- Multi-Agent 的三种典型结构与适用场景
- Multi-Agent 与 Workflow / Runtime / Controller / Planner 的关系
:::

前面我们讲了单个 Agent 的运行方式：它可以理解任务、制定计划、调用工具、观察结果，并根据反馈继续执行。

但现实中的复杂任务，往往不是一个 Agent 就能很好完成的。

比如你想做一个自动投放分析 Agent，它可能同时需要：

- 分析数据
- 判断账户问题
- 检查素材质量
- 生成优化建议
- 输出日报
- 必要时通知运营或投手

如果把所有能力都塞进一个 Agent，它会变得非常复杂：职责不清、上下文混乱、执行不可控、错误难定位。

这时就需要 Multi-Agent，多智能体系统。

简单来说：

Multi-Agent 是让多个 Agent 按不同角色分工协作，共同完成一个复杂任务的系统。

## 一、为什么需要 Multi-Agent？

单 Agent 更像是一个人干所有事。

它既要理解需求，又要拆解计划，还要查数据、调用工具、判断结果、写报告。

在简单场景下，这没问题。

但一旦任务变复杂，就容易出现几个问题。

### 1. 职责过重

比如一个 Agent 同时负责：

拉取数据  
分析异常  
判断原因  
生成策略  
输出汇报  
执行投放动作

这会导致它在每一轮推理中都要考虑太多事情。

结果就是：

- 判断链路变长
- 容易遗漏关键步骤
- 输出不稳定
- 出错后不好定位是哪里的问题

### 2. 上下文容易混乱

不同任务需要关注的上下文不一样。

例如：

数据分析 Agent 关心：

消耗、收入、ROI、CTR、CVR、postback、event10

素材分析 Agent 关心：

标题、封面、落地页、转化卖点、合规风险

投放策略 Agent 关心：

预算、出价、人群包、渠道、广告组结构

如果全部混在一个 Agent 里，模型很容易把不同上下文混合，导致判断不精确。

### 3. 复杂任务需要专业分工

在真实公司里，复杂项目通常不是一个人完成的，而是多个角色协作：

产品经理  
数据分析师  
研发工程师  
运营  
投手  
审核  
项目负责人

Multi-Agent 的思想，其实就是把这种组织协作方式，迁移到 AI 系统里。

每个 Agent 专注一类能力，最终由一个协调者把它们组织起来。

## 二、Multi-Agent 的核心思想

Multi-Agent 的核心不是多几个模型同时说话，而是：

把复杂任务拆成多个角色，每个 Agent 只负责自己擅长的部分，再通过协作机制组合成完整能力。

它通常包含三个关键点：

1. 角色分工
2. 协作协议
3. 结果汇总

## 三、典型结构：主管 Agent + 专家 Agent

最常见的 Multi-Agent 结构是：

Supervisor Agent + Expert Agents

也就是：

一个主管 Agent 负责统筹  
多个专家 Agent 负责具体任务

可以理解成一个小团队。

### 1. Supervisor Agent：主管智能体

主管 Agent 不一定亲自执行所有任务。

它主要负责：

理解用户目标  
拆解任务  
分配任务给不同专家 Agent  
收集各专家结果  
判断是否需要继续追问或重做  
汇总最终答案

它更像 Controller + Planner 的组合。

例如用户提出：

帮我分析昨天 Nutra 投放效果变差的原因，并给出优化建议。

Supervisor Agent 可能会拆成：

1. 让数据分析 Agent 检查 ROI、CTR、CVR 是否异常
2. 让流量质量 Agent 检查作弊率、渠道、bundle、tagid
3. 让投放策略 Agent 判断预算、出价、人群是否有问题
4. 让总结 Agent 生成对外汇报话术

### 2. Expert Agent：专家智能体

专家 Agent 只负责某个细分任务。

例如：

| Agent | 职责 |
| --- | --- |
| 数据分析 Agent | 拉数、算指标、找异常 |
| 流量质量 Agent | 判断假量、作弊、渠道质量 |
| 投放策略 Agent | 判断出价、预算、人群、广告组结构 |
| 素材分析 Agent | 分析素材点击率、卖点、疲劳度 |
| 报告生成 Agent | 把结果整理成日报、周报、客户话术 |
| 审核 Agent | 检查最终结果是否有逻辑错误和风险 |

这样每个 Agent 的上下文更干净，输出也更稳定。

## 四、典型模式一：主管型 Multi-Agent

主管型结构是最容易理解的一种。

结构如下：

用户  
↓  
主管 Agent  
↓  
专家 Agent A  
专家 Agent B  
专家 Agent C  
↓  
主管 Agent 汇总  
↓  
最终结果

适合什么场景？

适合任务比较复杂，但最终需要一个统一结论的场景。

比如：

投放日报分析  
股票交易综合判断  
竞品分析报告  
项目需求评审  
自动化运营诊断  
企业知识库问答

这类任务通常需要多个维度共同判断，不能只看单一数据。

### 举个广告投放例子

用户输入：

分析昨天 SOI 业务 ROI 下滑的原因。

主管 Agent 会先分配任务。

数据分析 Agent 负责看：

消耗是否上涨  
收入是否下降  
ROI 是否下降  
CTR 是否下降  
CVR 是否下降  
purchase 是否减少

流量质量 Agent 负责看：

是否有异常 affiliate  
是否有异常 bundle  
是否有异常 tagid  
click_cheat_ratio 是否升高  
imp_cheat_ratio 是否升高  
postback 是否异常

策略 Agent 负责看：

出价是否变化  
预算是否变化  
bid floor 是否变化  
渠道是否切换  
人群包是否变窄  
广告组是否进入衰退期

总结 Agent 负责输出：

问题结论  
核心原因  
建议动作  
对外话术  
下一步监控指标

最后主管 Agent 统一汇总。

## 五、典型模式二：流水线式 Agent 编排

第二种模式是流水线式。

也就是一个 Agent 的输出，作为下一个 Agent 的输入。

结构如下：

Agent A -> Agent B -> Agent C -> Agent D

这和传统 Workflow 很像，但区别是每个节点不是固定脚本，而是一个具备判断能力的 Agent。

适合什么场景？

适合步骤相对稳定，但每一步都需要智能判断的任务。

比如：

文章生成流程  
视频脚本生成流程  
广告素材生成流程  
代码生成流程  
数据报告生成流程

### 举个内容生产例子

假设你要做一个爆款文案改写 + 视频脚本生成系统。

流水线可以是：

热点识别 Agent  
↓  
文案拆解 Agent  
↓  
改写 Agent  
↓  
视频脚本 Agent  
↓  
审核 Agent  
↓  
发布建议 Agent

每一步只负责自己的工作。

例如：

文案拆解 Agent 只负责找出：

开头钩子  
痛点  
转折  
利益点  
行动号召

视频脚本 Agent 只负责把文案改成：

镜头  
画面  
口播  
字幕  
转场  
节奏

审核 Agent 只负责检查：

是否违规  
是否夸大  
是否逻辑不通  
是否适合平台发布

这就是典型的 Multi-Agent 流水线。

## 六、典型模式三：辩论式 / 评审式协作

第三种是 Debate / Review 模式。

也就是多个 Agent 从不同角度提出观点，然后再由一个 Agent 做最终裁决。

结构如下：

Agent A 提出观点  
Agent B 提出反对意见  
Agent C 补充风险  
Judge Agent 做最终判断

适合什么场景？

适合高风险、高不确定性的判断任务。

例如：

股票交易决策  
重大产品方案评审  
投放策略调整  
预算分配  
合规审核  
技术架构选型

因为这些场景不能只听一个角度。

### 举个股票判断例子

用户问：

今天 SOXL 是否适合低吸？

可以设计多个 Agent：

| Agent | 视角 |
| --- | --- |
| 技术面 Agent | 看支撑、压力、均线、MACD、RSI、成交量 |
| 板块 Agent | 看 SOXX、SMH、NVDA、AMD、AVGO |
| 大盘 Agent | 看 QQQ、纳指期货、美债、VIX |
| 风险 Agent | 判断是否追涨、是否假突破、是否需要止损 |
| 裁决 Agent | 综合所有观点，给出最终操作建议 |

这种结构比单 Agent 更稳，因为它会主动暴露分歧。

例如：

技术面 Agent：短线有反弹信号  
板块 Agent：半导体整体偏弱  
大盘 Agent：纳指期货未明显转强  
风险 Agent：不建议重仓低吸  
裁决 Agent：可以小仓试探，但不能满仓，必须设置止损

这就是评审式 Multi-Agent 的价值。

## 七、Multi-Agent 和 Workflow 的区别

很多人会把 Multi-Agent 和 Workflow 混在一起。

它们确实有相似点，但不是一回事。

### Workflow 更强调固定流程

Workflow 通常是：

步骤 A -> 步骤 B -> 步骤 C

流程提前设计好，每一步做什么比较固定。

例如：

拉数据 -> 清洗数据 -> 生成图表 -> 发送日报

这是一个标准自动化流程。

### Multi-Agent 更强调角色协作

Multi-Agent 不是只关心步骤顺序，而是关心谁负责什么判断。

例如：

数据 Agent 负责发现异常  
策略 Agent 负责解释原因  
风控 Agent 负责检查风险  
总结 Agent 负责生成话术

它更像一个智能团队，而不只是流水线。

## 八、Multi-Agent 和 Runtime / Controller / Planner 的关系

前面我们讲过：

Planner = 制定计划  
Controller = 控制流程  
Runtime = 执行动作  
Harness = 承载整个系统运行

放到 Multi-Agent 里，可以这样理解：

Planner：决定任务如何拆给不同 Agent  
Controller：调度不同 Agent 的执行顺序  
Runtime：为每个 Agent 提供工具和执行环境  
Harness：把整个多 Agent 系统托管起来

例如：

Supervisor Agent 发现需要查数据  
↓  
Controller 调用数据分析 Agent  
↓  
数据分析 Agent 使用 SQL Runtime 查询数据  
↓  
Runtime 返回结果  
↓  
Controller 把结果交给策略 Agent  
↓  
策略 Agent 生成优化建议  
↓  
Supervisor 汇总最终输出

所以 Multi-Agent 不是替代 Runtime、Planner、Controller，而是在它们之上组织出更复杂的协作结构。

## 九、Multi-Agent 的关键难点

Multi-Agent 听起来很强，但并不是 Agent 越多越好。

如果设计不好，反而会变得更慢、更乱、更难控。

主要难点有四个。

### 1. 角色边界不清

如果每个 Agent 的职责都很模糊，就会互相重复。

例如：

数据 Agent 也在给策略建议  
策略 Agent 也在重新算数据  
总结 Agent 又重新解释了一遍原因

这样会导致结果冲突。

所以设计 Multi-Agent 时，第一件事是明确每个 Agent 的职责边界。

### 2. 协作成本变高

Agent 越多，通信成本越高。

每个 Agent 都要输入、输出、等待、汇总。

如果任务很简单，强行使用 Multi-Agent 反而浪费。

例如：

帮我把这句话润色一下。

这种任务没必要启动多个 Agent。

单 Agent 就够了。

### 3. 结果冲突

多个 Agent 可能给出不同结论。

例如：

技术面 Agent 认为可以买  
风险 Agent 认为不能买  
大盘 Agent 认为要等确认

这时必须有一个裁决机制。

否则最终输出就会变成：

一方面可以买，另一方面也不能买。

这对用户没有价值。

### 4. 状态管理复杂

Multi-Agent 系统里，每个 Agent 都可能产生中间结果。

系统需要管理：

谁做过什么？  
结果是什么？  
是否可信？  
是否需要重试？  
哪些结果需要传递给下一个 Agent？  
哪些结果应该丢弃？

这就是为什么 Harness 和 Controller 很重要。

它们负责让多 Agent 系统有序运行，而不是一群 Agent 混乱对话。

## 十、什么时候应该使用 Multi-Agent？

不是所有任务都需要 Multi-Agent。

可以用下面这个判断标准。

### 适合使用 Multi-Agent 的场景

当任务满足以下特征时，适合使用 Multi-Agent：

1. 任务复杂，需要多个专业视角
2. 输出结果需要评审或校验
3. 每个子任务都有明显不同的能力要求
4. 需要长期运行和状态管理
5. 出错成本较高，需要多层检查

例如：

企业级投放 Agent  
自动数据分析系统  
复杂客服系统  
投资决策辅助系统  
代码开发 Agent  
内容生产流水线  
合规审核系统

### 不适合使用 Multi-Agent 的场景

如果任务本身很简单，就不适合。

例如：

一句话润色  
简单翻译  
单次 SQL 改写  
普通问答  
简单总结

这些任务用单 Agent 更直接、更快、更稳定。

## 十一、一个企业级 Agent 架构示例

假设我们要做一个企业级广告投放 Agent，可以这样设计：

用户输入  
↓  
Supervisor Agent  
↓  
任务拆解  
↓  
数据分析 Agent  
流量质量 Agent  
素材分析 Agent  
投放策略 Agent  
合规审核 Agent  
日报生成 Agent  
↓  
结果汇总  
↓  
最终输出

每个 Agent 的职责如下：

| Agent | 核心职责 |
| --- | --- |
| Supervisor Agent | 理解目标、拆任务、分发任务、汇总结果 |
| 数据分析 Agent | 查询消耗、收入、ROI、CTR、CVR、转化漏斗 |
| 流量质量 Agent | 分析作弊率、异常 IP、bundle、tagid、渠道质量 |
| 素材分析 Agent | 分析素材点击率、转化率、疲劳度、卖点 |
| 投放策略 Agent | 输出预算、出价、人群、渠道、广告组优化建议 |
| 合规审核 Agent | 检查是否涉及违规表达、客户敏感信息、错误承诺 |
| 日报生成 Agent | 生成飞书日报、客户汇报、复盘总结 |

这个系统比单 Agent 更接近真实企业协作。

## 十二、Multi-Agent 的本质

Multi-Agent 的本质不是让多个 AI 聊天，而是：

用组织结构解决复杂任务。

单 Agent 解决的是一个智能体如何完成任务。

Multi-Agent 解决的是多个智能体如何分工、协作、校验和汇总。

它的核心价值是：

让复杂任务可拆解  
让专业能力可复用  
让结果可以互相校验  
让系统更接近真实组织运转方式

## 总结

Multi-Agent 多智能体系统，是 Agent 从单人执行走向团队协作的关键阶段。

它通常由一个主管 Agent 负责任务拆解和结果汇总，再由多个专家 Agent 分别处理数据、策略、审核、执行等不同环节。

在简单任务中，单 Agent 就足够；但在企业级场景、复杂分析场景、自动化运营场景中，Multi-Agent 能显著提升系统的稳定性、可解释性和可扩展性。

可以用一句话记住：

单 Agent 是一个人解决问题，Multi-Agent 是一个团队解决问题。

真正有价值的 Multi-Agent 系统，不是 Agent 数量多，而是分工清晰、协作稳定、结果可控。
