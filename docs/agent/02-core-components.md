:::tip 先看结论
**Agent 不是“大模型 + Prompt”，而是一个由四个模块协同工作的系统：LLM、Memory、Tools、Planner / Controller。**

读完这篇你会明确三件事：
- 每个模块分别负责什么，以及为什么缺一不可
- 为什么只有 LLM 无法完成真实业务任务
- 一个可生产化 Agent 应该如何分层协同与权限控制
:::

上一篇文章里，我们讲了为什么 AI 会从 Chatbot 走向 Agent。

核心原因是：真实工作不是一次问答，而是一条连续任务链路。

一个 Agent 要想真正完成任务，不能只依赖大模型本身。大模型只是 Agent 的核心能力之一，但不是全部。

一个完整的 Agent 系统，通常由四类核心模块组成：

**LLM**（负责理解、推理、生成） -> **Memory**（负责保存上下文、经验和用户偏好） -> **Tools**（负责连接外部系统，获得执行能力） -> **Planner / Controller**（负责拆解任务、调度流程、控制执行）

如果用一句话概括：

LLM 负责“想和说”，Memory 负责“记住”，Tools 负责“做事”，Planner / Controller 负责“安排怎么做”。

## 一张图看懂 Agent 的基本结构

可以先看一个简化版结构图：

**用户输入目标**  
-> **Planner / Controller（任务规划与流程控制）**  
-> 并行调用：**LLM（理解、推理、生成）** + **Memory（短期/长期记忆）** + **Tools（数据库/API/搜索/文件/浏览器）**  
-> **Planner / Controller 汇总与决策**  
-> **输出结果（回答 / 报告 / 执行动作）**

这个图表达的是：

Agent 不是一个单独的大模型，而是一个由多个模块组成的系统。

用户提出目标后，Controller 会决定下一步该做什么；LLM 负责理解和推理；Memory 提供历史上下文；Tools 负责连接真实世界的系统；最后 Controller 汇总结果并输出。

## 为什么不能只靠 LLM

很多人刚开始理解 Agent 时，会误以为：

Agent = 大模型 + Prompt

这种理解是不完整的。

大模型确实很重要，但如果只有大模型，它仍然有明显限制：

- 不知道最新数据
- 不能直接查询数据库
- 不能操作外部系统
- 不能长期记住用户偏好
- 不能稳定管理复杂流程
- 不能天然保证每一步都可控

比如你问一个普通大模型：

帮我分析昨天 847 广告组为什么消耗下降。

如果它没有连接数据库，它只能基于经验猜测：

- 可能是预算不足
- 可能是出价偏低
- 可能是流量竞争加剧
- 可能是素材点击率下降
- 可能是转化率降低

这些回答听起来合理，但没有数据支撑。

一个真正的投放 Agent 应该怎么做？

1. 先理解用户要分析的是 847
2. 判断需要查询昨天和对比周期的数据
3. 调用 SQL 工具查询消耗、曝光、点击、转化、ROI
4. 按渠道、广告组、bundle、人群包拆分
5. 找出下降最明显的维度
6. 判断是流量问题、竞价问题、转化问题还是审核问题
7. 输出结论和建议

这就是 Agent 和普通 LLM 的区别。

普通 LLM 主要负责“说出可能原因”。

Agent 要负责“查证原因并推进任务”。

## 组件一：LLM

LLM，全称是 Large Language Model，也就是大语言模型。

在 Agent 系统里，LLM 是最核心的智能能力来源。

它主要负责：

- 理解用户输入
- 分析任务意图
- 进行逻辑推理
- 生成文本内容
- 总结工具返回结果
- 决定下一步可能动作

你可以把 LLM 理解为 Agent 的“大脑能力”。

但注意，LLM 更像是“大脑中的语言和推理部分”，不是整个 Agent。

### LLM 在 Agent 中负责什么

LLM 最常见的职责有几类。

#### 理解目标

用户通常不会把需求描述得非常标准。

比如用户可能说：

帮我看一下 847 怎么回事，今天量掉了。

这句话里有很多隐含信息：

- 847 可能是某个广告主、campaign 或 ad group
- “量掉了”可能指消耗下降，也可能指曝光下降、订单下降
- “今天”需要转换成具体日期
- 需要和昨天、近 7 日或上周同期对比

LLM 的作用就是把自然语言变成更明确的任务目标。

比如理解为：

分析 847 今日广告消耗下降原因，并与昨日和近 7 日均值对比，输出异常原因和优化建议。

#### 推理判断

LLM 还负责根据已有信息做判断。

比如工具返回数据后：

- 曝光下降 40%
- 点击率基本稳定
- 点击到转化率下降不明显
- 预算未耗尽
- 平均出价下降
- winrate 下降明显

LLM 可以推理出：

主要问题可能不是素材点击率，也不是落地页转化，而是竞价竞争力下降导致拿量减少。

这就是推理能力。

#### 生成输出

Agent 最终需要把结果表达出来。

比如输出：

- 分析报告
- 日报
- 客户话术
- 飞书播报
- SQL 解释
- 异常排查结论
- 操作建议

这些都依赖 LLM 的语言生成能力。

例如：

今日 847 消耗下降主要由曝光规模减少导致，CTR 和后链路转化率波动不大，初步判断核心问题在于竞价竞争力下降。建议优先检查出价策略和预算分配，同时保留核心出单人群包投放，控制探索流量预算占比。

这类表达就是 LLM 擅长的部分。

### LLM 不适合单独负责什么

LLM 很强，但不能让它什么都做。

尤其在 Agent 系统里，以下事情不应该完全交给 LLM 自由发挥：

- 直接执行高风险操作
- 直接修改预算
- 直接关闭广告组
- 直接调用生产接口
- 直接决定资金交易
- 直接跳过数据验证
- 直接生成不受限制的 SQL

原因很简单：

LLM 会推理，但它不是天然可靠的执行控制系统。

它可能理解错字段，可能遗漏条件，可能把相似概念混淆，也可能生成看似合理但实际错误的结论。

所以在严肃业务系统里，LLM 应该被放在一个有边界的位置：

- 可以理解
- 可以分析
- 可以建议
- 可以生成

但关键执行必须被 Controller、规则、权限和人工审批约束。

## 组件二：Memory

Memory 是 Agent 的记忆系统。

如果没有 Memory，Agent 每次对话都像第一次见你。

它不知道：

- 你是谁
- 你在做什么业务
- 你常用哪些指标
- 你之前分析过什么问题
- 你有哪些偏好
- 哪些策略之前成功过
- 哪些策略之前失败过

这样的 Agent 很难长期协作。

Memory 的作用，就是让 Agent 能够保存和复用上下文。

### Memory 分为短期记忆和长期记忆

Agent 的 Memory 通常可以分成两类：

短期记忆  
↓  
当前任务中的上下文  
长期记忆  
↓  
跨任务、跨会话、长期保存的经验和偏好

### 短期记忆

短期记忆主要用于当前任务。

比如你正在让 Agent 分析一个广告问题：

第一轮：帮我看 847 今天消耗为什么下降  
第二轮：再按 ad_group 拆一下  
第三轮：看看是不是 CPE 掉量  
第四轮：帮我生成一段内部同步话术

这里的 Agent 必须记住前面几轮的信息：

- 分析对象是 847
- 问题是消耗下降
- 已经看过整体数据
- 现在要继续按 ad_group 和 CPE 维度拆分
- 最终要生成内部话术

这就是短期记忆。

短期记忆的重点是：

围绕当前任务保持上下文连续。

### 长期记忆

长期记忆则是跨任务保存的信息。

比如一个投放 Agent 长期服务你，它可以记住：

- 你常看的核心指标是 ROI、event10、CTR、CVR、winrate
- 你常分析的行业有 Nutra、SOI、HotSku、杀毒、保险
- 你常用的维度有 adv_id、ad_group_id、bundle、domain、channel
- 你偏好输出“原因 + 数据依据 + 建议 + 话术”
- 你不希望系统自动执行高风险动作
- 你常用 Spark SQL 和 StarRocks 查询数据

这些信息不会只在一次任务中有用，而是会长期提升 Agent 的协作效率。

长期记忆的重点是：

沉淀用户偏好、业务规则、历史经验和成功案例。

### Memory 对 Agent 的价值

Memory 不是简单地“保存聊天记录”。

真正有价值的 Memory 应该帮助 Agent 做三件事。

#### 复用上下文

如果 Agent 记住了你的业务背景，你就不用每次都解释一遍。

比如你说：

看下 847 今天为什么掉量。

Agent 就能知道你大概率需要看：

- 消耗
- 曝光
- 点击
- 转化
- ROI
- event10
- 出价
- winrate
- 人群包
- ad_group
- bundle

而不是泛泛回答。

#### 沉淀经验

Agent 应该能记住过去发生过什么。

比如：

- 某个渠道之前假量较多
- 某类 bundle 转化质量差
- 某个客户对 Google 审核比较敏感
- 某个行业更适合先跑 CPE 再保留 CPC 对照

这些经验可以帮助 Agent 在未来分析时给出更接近真实业务的判断。

#### 个性化输出

不同用户需要的输出风格不同。

有的人要技术报告。  
有的人要老板能看懂的总结。  
有的人要客户话术。  
有的人要 SQL。  
有的人要飞书播报。

Memory 可以记住你的偏好，让 Agent 输出更贴合你的使用习惯。

### Memory 的风险

Memory 也不能无限保存。

一个好的 Agent 系统必须处理好记忆边界。

常见风险包括：

- 记住了错误信息
- 记住了过期策略
- 记住了敏感数据
- 把短期信息误当成长期偏好
- 引用历史经验时没有重新验证

比如某个 bundle 过去表现好，不代表今天还表现好。

所以 Memory 不能替代实时数据。

更合理的做法是：

- Memory 提供历史参考
- Tools 提供实时数据
- LLM 负责综合判断
- Controller 负责决定是否执行

## 组件三：Tools

Tools 是 Agent 的外部能力系统。

如果说 LLM 让 Agent 能“思考和表达”，那么 Tools 让 Agent 能“连接真实世界并采取行动”。

没有 Tools，Agent 只能停留在文本层面。

有了 Tools，Agent 才能真正完成任务。

### Tools 可以是什么

Tools 的范围很广。

常见工具包括：

- 搜索引擎
- 数据库
- API
- 代码解释器
- 浏览器
- 文件系统
- 邮件
- 飞书 / Slack
- Notion
- GitHub
- CRM
- 广告投放平台
- 数据看板
- 支付系统
- 工单系统

对于投放 Agent 来说，Tools 可能包括：

- SQL 查询工具
- DSP 数据接口
- 广告平台 API
- 反作弊系统
- 素材库
- 人群包系统
- 预算调整接口
- 飞书通知机器人
- 日报生成工具

这些工具决定了 Agent 能做什么。

### Tools 让 Agent 从“回答”变成“执行”

举个例子。

用户问：

昨天 SOI 的 ROI 为什么下降了？

没有 Tools 的 AI 只能回答：

可能是转化率下降、流量质量下降、素材疲劳、出价变化等原因。

有 Tools 的 Agent 可以这样做：

1. 查询昨天 SOI 整体 ROI
2. 对比前一天和近 7 日均值
3. 按渠道拆分 ROI
4. 按 ad_group 拆分 ROI
5. 检查 click、lp_click、purchase、event10 漏斗
6. 查询异常 bundle 和 domain
7. 判断是否存在假量或低质流量
8. 输出明确原因和建议

这就不是泛泛分析，而是基于真实数据执行任务。

### Tool Calling 的本质

Tool Calling 的本质是：

- LLM 决定要调用什么工具
- 系统按照结构化参数执行工具
- 工具返回结果
- LLM 再根据结果继续判断

一个简化流程是：

用户：帮我查 847 昨天消耗  
LLM 判断：  
需要调用 SQL 查询工具  
Tool 执行：  
查询数据库  
Tool 返回：  
昨天消耗 1300，转化 24，ROI 0.92  
LLM 总结：  
昨天 847 消耗 1300，整体 ROI 0.92，建议继续拆分 ad_group 查看差异。

这就是 Agent 执行任务的基础。

### Tools 的边界很重要

Tools 让 Agent 变强，但也带来风险。

尤其是当 Tools 可以执行真实操作时，比如：

- 修改预算
- 关闭广告组
- 调整出价
- 发送邮件
- 删除文件
- 发起支付
- 提交代码
- 修改数据库

这些都必须设置边界。

一个成熟 Agent 系统通常会把工具分成几类：

| 工具类型 | 风险级别 | 是否需要审批 |
| --- | --- | --- |
| 查询类工具 | 低 | 通常不需要 |
| 分析类工具 | 低 | 通常不需要 |
| 生成类工具 | 中 | 视场景而定 |
| 通知类工具 | 中 | 重要消息需要确认 |
| 修改类工具 | 高 | 建议必须审批 |
| 支付/资金/生产变更类工具 | 极高 | 必须审批 |

对于投放 Agent 来说，合理边界应该是：

- 可以自动查数
- 可以自动分析
- 可以自动生成建议
- 可以自动生成话术
- <strong style={{color: 'var(--ifm-color-primary)'}}>但不能未经确认直接调预算、关广告组、改出价</strong>

这就是工具权限控制。

## 组件四：Planner / Controller

Planner / Controller 是 Agent 的任务规划与流程控制模块。

如果 LLM 是推理能力，Memory 是记忆能力，Tools 是执行能力，那么 Planner / Controller 就是整个 Agent 的调度中心。

它负责决定：

- 下一步做什么
- 调用哪个工具
- 是否需要继续查询
- 是否需要人工确认
- 是否终止任务
- 是否输出最终结果

### Planner 和 Controller 的区别

严格来说，Planner 和 Controller 可以区分开。

Planner 更偏“计划”：

- 把目标拆成步骤
- 决定先做什么后做什么
- 制定执行路径

Controller 更偏“控制”：

- 调度每一步执行
- 管理状态
- 处理失败
- 控制权限
- 决定是否进入下一步

在很多 Agent 系统里，这两个能力会合在一起，所以我们常写成：

Planner / Controller

### Planner 负责拆解任务

比如用户说：

帮我做一份昨天 Nutra 的投放复盘。

Planner 需要拆解为：

1. 查询 Nutra 昨日整体数据
2. 对比前一日和近 7 日均值
3. 按 adv_id 拆分
4. 按 ad_group 拆分
5. 按渠道拆分
6. 分析漏斗：imp -> click -> lp_click -> purchase
7. 识别异常项
8. 给出优化建议
9. 生成复盘文案

这一步决定了 Agent 的工作路径。

如果 Planner 拆错了，后面即使工具调用正确，结果也可能偏离目标。

### Controller 负责执行控制

Controller 更像一个流程引擎。

它需要管理：

- 当前执行到哪一步
- 上一步结果是什么
- 下一步是否继续
- 工具调用是否成功
- 是否需要重试
- 是否需要人工确认
- 是否达到终止条件

比如执行 SQL 时失败了：

- 字段不存在
- 权限不足
- 数据为空
- 日期格式错误

Controller 应该决定：

- 是否让 LLM 修正 SQL
- 是否换一张表查询
- 是否提示用户权限问题
- 是否跳过该维度
- 是否终止任务

这就是控制能力。

### 为什么 Planner / Controller 很关键

很多 Agent 失败，不是因为模型不聪明，而是因为流程不可控。

常见问题包括：

- 任务没有拆清楚
- 工具调用顺序混乱
- 查到数据后没有继续分析
- 无限循环调用工具
- 没有判断任务是否完成
- 高风险操作没有审批
- 失败后不会恢复
- 输出结果没有结构

Planner / Controller 的作用就是解决这些问题。

尤其在企业级 Agent 中，Controller 比单纯的 Prompt 更重要。

因为企业系统需要的是：

- 稳定
- 可控
- 可审计
- 可复盘
- 可回滚

而不是让模型自由发挥。

## 四个组件如何协同工作

现在我们把四个组件放在一起，看一个完整流程。

用户输入：

帮我分析 847 昨天消耗下降原因，并给一段内部同步话术。

Agent 执行过程可能是：

1. Controller 接收任务
2. LLM 理解目标：分析 847 昨日消耗下降原因
3. Memory 提供背景：847 核心投放 CPE，一方人群和出单人群包为主
4. Planner 拆解任务：先查整体，再拆 ad_group、渠道、人群包、bundle
5. Tools 查询数据：消耗、曝光、点击、转化、ROI、winrate
6. LLM 分析工具返回结果
7. Controller 判断是否需要继续追加查询
8. Tools 继续查询异常维度
9. LLM 生成原因判断和优化建议
10. Controller 检查是否涉及高风险执行动作
11. 最终输出内部同步话术

这个过程里，每个组件都有自己的职责：

| 组件 | 在任务中的作用 |
| --- | --- |
| LLM | 理解问题、分析结果、生成话术 |
| Memory | 提供历史背景和用户偏好 |
| Tools | 查询真实数据 |
| Planner / Controller | 拆解任务、控制流程、决定下一步 |

这就是 Agent 的基本工作方式。

## 用投放 Agent 举一个完整例子

假设你要做一个广告投放 Agent。

它的核心目标是：

帮助投放团队发现问题、分析原因、给出建议、生成汇报，并在人工确认后执行部分操作。

它可以被设计成下面这样：

**用户需求（看下今天为什么掉量）**  
-> **Controller 识别任务类型**  
-> 读取 **Memory（业务背景）** + 调用 **SQL Tool（投放数据）** + 调用 **Anti-Fraud Tool（假量风险）**  
-> **LLM 分析与归因**  
-> **生成策略建议（预算 / 出价 / 人群 / 渠道）**  
-> **判断是否高风险操作**  
-> 低风险：自动生成日报/话术  
-> 高风险：人工确认后再执行 Tool（调整预算/出价）

在这个系统里：

- LLM 不直接掌控一切
- Memory 不替代实时数据
- Tools 不无边界开放
- Controller 不让流程失控

这才是一个更接近生产环境的 Agent 设计。

## 一个常见误区：Agent 不是越自主越好

很多人以为 Agent 越自主越好。

比如：

- 自动分析
- 自动决策
- 自动执行
- 自动优化
- 自动复盘

听起来很美好，但在真实业务里很危险。

一个不受控制的 Agent 可能会：

- 误读数据
- 错误调用接口
- 扩大错误策略
- 误关广告组
- 误发客户消息
- 把临时结论当成最终结论

所以一个好的 Agent，不是越自由越好，而是越可控越好。

尤其在投放、金融、医疗、法务、生产系统这类场景中，更应该采用：

- 低风险自动化
- 高风险人工确认
- 全流程留痕
- 关键节点可回滚

Agent 的目标不是替人失控地做决定，而是在人的边界内高效完成任务。

## Agent 的核心能力模型

我们可以把 Agent 的核心能力总结为一个公式：

Agent = LLM + Memory + Tools + Planner / Controller

其中：

- LLM 提供智能理解和推理
- Memory 提供上下文和经验
- Tools 提供外部执行能力
- Planner / Controller 提供流程管理和安全边界

如果缺少其中任何一个，Agent 的能力都会不完整。

只有 LLM，没有 Tools：

只能回答，不能执行。

只有 Tools，没有 LLM：

只是普通自动化脚本。

只有 LLM 和 Tools，没有 Controller：

容易流程失控，难以生产化。

只有短期上下文，没有长期 Memory：

每次任务都要从零开始。

所以，Agent 不是一个单点能力，而是一个系统工程。

## 结论：Agent 是一个由多个模块协同工作的系统

理解 Agent，不能只盯着大模型。

大模型是 Agent 的核心，但不是 Agent 的全部。

一个真正可用的 Agent，需要同时具备：

- 理解和推理能力
- 记忆和上下文能力
- 工具调用和执行能力
- 任务规划和流程控制能力

也就是说：

LLM 让 Agent 能理解。  
Memory 让 Agent 能记住。  
Tools 让 Agent 能行动。  
Planner / Controller 让 Agent 能有序完成任务。

这四个模块共同构成了 Agent 的基本骨架。

理解了这套结构，再去看 LangGraph、Google ADK、OpenAI Agents SDK、OpenClaw、Hermes 这些框架时，就不会只看表面名词，而是能判断它们分别在解决 Agent 系统中的哪一部分问题。

Agent 是如何运行的。

也就是一个 Agent 从接收目标，到拆解任务、调用工具、观察结果、修正计划、最终输出结果，完整经历了什么过程。
