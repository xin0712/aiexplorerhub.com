:::tip 先看结论
Memory 不是让 Agent 记住一切，而是让它在正确的时间，拿出正确的历史信息来完成当前任务。

这篇你会理解三件事：

- 为什么没有记忆，Agent 只能停留在单轮问答
- 长期记忆系统的核心闭环：写入、存储、检索、更新（以及遗忘）
- 如何避免记忆污染，让 Agent 长期可控、可用、可维护
:::

在前面我们讲了 Agent 是如何运行的：它不是一次性回答问题，而是通过“理解任务 -> 制定计划 -> 调用工具 -> 观察结果 -> 继续推理”的循环来完成复杂任务。

但是，如果 Agent 每次运行都像“失忆”一样，只能看到当前输入，那么它就很难真正变得智能。

一个真正可用的 Agent，必须具备记忆能力。

记忆系统决定了 Agent 能否理解上下文、复用历史经验、记住用户偏好，并在多轮任务中保持一致的行为。

这一章我们重点拆解 Agent 的 Memory 系统。
## 一、为什么 Agent 需要记忆？

普通大模型本身并不天然拥有长期记忆。

你给它一个问题，它根据当前上下文生成回答。下一次对话，如果历史信息没有被放进上下文，它就不知道之前发生过什么。

这会带来几个问题：

第一，Agent 无法持续理解用户。

比如用户之前说过：

我是做广告投放系统的产品经理，主要关注 DSP、素材平台、自动化投放。

如果 Agent 不能记住这个信息，那么每次用户问投放系统、广告策略、数据看板时，它都要重新解释背景，效率很低。

第二，Agent 无法积累经验。

比如某个广告账户之前测试过 Vungle 流量效果不错，Taboola 某些 domain 表现较差。如果 Agent 不能记住这些历史实验结果，下次做投放建议时就无法复用经验。

第三，Agent 无法完成长周期任务。

比如用户让 Agent 帮他连续一周监控 SOXL、SOXS 的盘前走势。如果没有记忆系统，Agent 就不知道前几天的判断结果、执行策略和盈亏反馈。

所以，Agent 的记忆系统，本质上解决的是：

如何让 Agent 不只是“当前这一轮聪明”，而是能够在长期任务中持续积累、检索和更新信息。
## 二、Agent 记忆可以分成两类

Agent 的记忆通常可以分成两类：

1. 短期记忆
2. 长期记忆

这两个概念可以类比人类。

短期记忆就像你现在脑子里正在处理的信息，比如刚刚听到的一句话、当前正在看的页面、正在分析的一组数据。

长期记忆则像你过去积累下来的知识、经验、偏好和历史事件。

对应到 Agent 系统中：

短期记忆 = 当前上下文窗口  
长期记忆 = 外部存储系统
## 三、短期记忆：上下文窗口管理

短期记忆主要指模型当前能看到的上下文。

比如一次对话里，用户连续说了几句话：

用户：帮我分析一下 SOXL 今天适不适合买。  
用户：重点看半导体板块和纳指期货。  
用户：如果盘前已经涨太多，就不要追高。

这三句话都会进入当前上下文。模型在回答时，会综合这些信息。

这就是短期记忆。

它的特点是：

优点：实时、直接、相关性强  
缺点：容量有限、容易被截断、无法永久保存

大模型都有上下文长度限制。即使现在模型上下文越来越长，也不代表可以无限塞入历史信息。

因为上下文越长，成本越高，噪音越多，模型越容易抓不住重点。

所以，Agent 不能简单地把所有历史内容都塞进 prompt，而是需要做上下文窗口管理。

### 1. 上下文窗口里应该放什么？

一个好的 Agent，不是把所有信息都放进去，而是只放当前任务最需要的信息。

通常包括：

- <span style={{color: 'var(--ifm-color-primary)'}}><strong>当前用户问题</strong></span>
- <span style={{color: 'var(--ifm-color-primary)'}}><strong>最近几轮对话</strong></span>
- <span style={{color: 'var(--ifm-color-primary)'}}><strong>当前任务目标</strong></span>
- <span style={{color: 'var(--ifm-color-primary)'}}><strong>关键约束条件</strong></span>
- <span style={{color: 'var(--ifm-color-primary)'}}><strong>相关工具返回结果</strong></span>
- <span style={{color: 'var(--ifm-color-primary)'}}><strong>必要的历史记忆摘要</strong></span>

比如用户问：

帮我写一段广告投放复盘话术。

Agent 不需要把用户所有历史聊天都放进上下文，只需要知道：

- 用户是广告投放/产品经理
- 关注 DSP、渠道、ROI、CVR、反作弊
- 偏好简洁、客户可读的话术
- 本次复盘涉及哪个行业、哪个渠道、哪些数据

这就是上下文窗口管理的核心：

不是记得越多越好，而是当前任务需要什么，就取什么。
## 四、长期记忆：外部存储系统

长期记忆指的是 Agent 在当前上下文之外，额外保存的信息。

这些信息通常不会一直放在 prompt 里，而是存在外部系统中。

常见存储方式包括：

- 数据库
- 向量库
- 文档库
- KV 存储
- 用户画像系统
- 任务日志系统
- 知识库

长期记忆可以保存很多类型的信息。

例如：

用户偏好：  
用户喜欢简洁、直接、结构化的回答。

用户背景：  
用户是广告投放系统方向的产品经理。

历史项目：  
用户做过自动化投放平台、素材平台、代理结算系统。

业务经验：  
Nutra、SOI、HotSku、杀毒、保险是用户常分析的业务类型。

历史任务：  
上次已经帮用户设计过一个 SOXL/SOXS 的盘前判断框架。

这些信息不一定每次都出现，但当用户提出相关问题时，Agent 可以把它们检索出来，重新放进上下文。
## 五、长期记忆的核心流程

一个完整的长期记忆系统，一般包含四个动作：

- 写入
- 存储
- 检索
- 更新

也可以理解为：

- 什么信息值得记？
- 记到哪里？
- 什么时候取出来？
- 旧信息如何修改？

### 1. 写入：什么信息值得被记住？

不是所有信息都应该写入长期记忆。

比如用户随口说：

我现在有点饿。

这通常不值得长期保存。

但如果用户说：

以后你帮我写 SQL 的时候，尽量给完整代码，不要只给片段。

这就值得保存，因为它会影响未来很多次回答。

适合写入长期记忆的信息通常有几类：

- 长期稳定的用户偏好
- 长期有效的用户背景
- 反复出现的业务信息
- 历史项目经验
- 用户明确要求记住的信息
- 长期任务的阶段性进展

不适合写入长期记忆的信息包括：

- 一次性的临时情绪
- 短期无效的上下文
- 敏感隐私信息
- 没有未来使用价值的闲聊
- 不确定、未经确认的信息

所以，记忆系统的第一步不是“存”，而是“判断是否值得存”。

### 2. 存储：记忆应该怎么存？

长期记忆不能只是把原文完整堆进去。

如果只是把所有聊天原文存起来，后面检索会非常混乱。

更好的方式是把记忆结构化。

例如原始对话是：

用户：以后帮我写广告复盘的时候，不要太官方，要像我发给客户看的微信消息，简单直接一点。

可以被整理成记忆：

```json
{
  "type": "preference",
  "topic": "广告复盘话术",
  "content": "用户偏好客户可读、微信消息风格、简单直接的话术，不喜欢过于官方。",
  "confidence": "high",
  "updated_at": "2026-05-27"
}
```

这样做的好处是：

- 方便检索
- 方便更新
- 方便去重
- 方便判断适用场景

Agent 的记忆不应该只是“聊天记录仓库”，而应该是“可检索、可更新、可治理的知识资产”。
## 六、检索：如何从记忆中找到当前有用的信息？

长期记忆只有被正确检索出来，才有价值。

否则，记得再多也没有用。

常见的检索方式有三种：

### 1. 关键词检索

关键词检索适合查明确字段。

比如用户问：

SOI 业务有哪些 adv_id？

系统可以通过关键词：

- SOI
- adv_id

找到对应记忆。

优点是精准，缺点是语义理解能力弱。

### 2. 向量检索

向量检索适合查语义相似内容。

比如用户问：

帮我写一个客户能看懂的投放复盘。

虽然这句话里没有出现“广告话术偏好”，但向量检索可以找到类似记忆：

用户偏好广告复盘话术简洁、客户可读、不要太官方。

这就是向量检索的价值。

它不是只看字面关键词，而是看语义相似度。

### 3. 混合检索

实际 Agent 系统中，最常用的是混合检索：

关键词检索 + 向量检索 + 规则过滤

比如：

- 先用关键词筛出业务相关记忆
- 再用向量相似度排序
- 最后根据时间、置信度、任务场景过滤

这样可以兼顾准确率和召回率。
## 七、更新：记忆不是一成不变的

记忆系统最容易被忽视的问题是：记忆会过期。

比如用户之前说：

我主要关注课后补习创业。

后来用户又说：

我现在重点在做广告投放系统和美股交易工具。

如果系统还一直拿旧记忆来回答，就会产生记忆污染。

所以长期记忆必须支持更新。

更新策略包括：

- 覆盖旧记忆
- 追加新版本
- 降低旧记忆权重
- 标记为过期
- 保留历史但不主动调用

例如：

```json
{
  "memory": "用户正在做课后补习创业",
  "status": "deprecated",
  "reason": "用户近期重点转向广告投放系统和交易工具",
  "updated_at": "2026-05-27"
}
```

好的记忆系统不是只会“记住”，还要会“修正”。
## 八、遗忘：Agent 也需要学会忘记

很多人一提到记忆系统，就只想到“多存一点”。

但真正成熟的记忆系统，必须有遗忘机制。

因为不遗忘会带来几个问题：

- 记忆越来越多，检索噪音越来越大
- 过期信息干扰当前判断
- 敏感信息长期留存带来风险
- 用户不再认可旧信息
- 系统成本持续上升

遗忘不一定是物理删除，也可以是降低权重。

常见方式有：

- 用户主动要求删除
- 信息长期未使用
- 信息被新记忆覆盖
- 信息置信度较低
- 信息与当前事实冲突
- 信息属于短期任务状态

比如用户之前让 Agent 记住一个临时项目，但项目已经结束，那么这类记忆就不应该继续强影响后续回答。
## 九、记忆污染：最危险的问题

Memory 系统最大的问题之一是记忆污染。

所谓记忆污染，就是 Agent 记住了错误的、不完整的、过期的或者不该记的信息，并在未来回答中反复使用。

常见情况包括：

- 把用户的一次性想法当成长期偏好
- 把未确认的信息当成事实
- 把旧项目当成当前项目
- 把错误总结写入记忆
- 把敏感信息用于不该使用的场景

比如用户只是说：

我今天想试试 SOXS。

系统不能因此记成：

用户长期偏好做空半导体。

这就是错误泛化。

所以记忆写入时要控制粒度：

- 不要过度推断
- 不要扩大结论
- 不要把临时行为变成长期画像
- 不要把猜测当事实

一句话：

记忆系统不是越聪明越好，而是越克制越可靠。
## 十、一个 Agent Memory 的基本架构

一个可用的 Agent 记忆系统，大致可以设计成这样：

用户输入  
  ↓  
判断是否需要检索记忆  
  ↓  
从短期上下文中提取当前任务  
  ↓  
从长期记忆中检索相关信息  
  ↓  
筛选、排序、去重  
  ↓  
将相关记忆注入 prompt  
  ↓  
模型生成回答  
  ↓  
判断本轮是否产生新记忆  
  ↓  
写入 / 更新 / 忽略

可以简化成：

Memory = 记什么 + 怎么存 + 怎么找 + 怎么改 + 怎么忘
## 十一、短期记忆和长期记忆如何配合？

短期记忆负责当前任务，长期记忆负责历史积累。

二者的关系可以这样理解：

短期记忆：我现在正在做什么？  
长期记忆：我过去知道什么？

比如用户问：

帮我优化这段广告复盘话术。

短期记忆里有：

- 用户当前给出的原始话术
- 本轮要求：优化表达

长期记忆里有：

- 用户偏好简洁、客户可读
- 用户常做 DSP、渠道、ROI、CVR 分析
- 用户不喜欢过于官方的话术

Agent 最终回答时，就会结合两部分：

- 基于当前这段话术进行改写
- 同时使用用户长期偏好的表达风格

这才是记忆系统真正的价值。
## 十二、Memory 系统的设计原则

最后总结一下，Agent 的记忆系统有几个关键原则。

<span style={{color: 'var(--ifm-color-primary)'}}><strong>第一，记忆要服务任务，而不是为了存储而存储。</strong></span>  
<span style={{color: 'var(--ifm-color-primary)'}}><strong>第二，长期记忆必须结构化，否则后期不可维护。</strong></span>  
<span style={{color: 'var(--ifm-color-primary)'}}><strong>第三，检索要精确，不能把不相关记忆强行塞进上下文。</strong></span>  
<span style={{color: 'var(--ifm-color-primary)'}}><strong>第四，记忆要有更新时间和置信度。</strong></span>  
<span style={{color: 'var(--ifm-color-primary)'}}><strong>第五，要允许更新和遗忘，不能让旧信息长期污染新判断。</strong></span>  
<span style={{color: 'var(--ifm-color-primary)'}}><strong>第六，不要过度推断用户，只记录有明确依据的信息。</strong></span>  
<span style={{color: 'var(--ifm-color-primary)'}}><strong>第七，敏感信息要谨慎处理，不能随便写入长期记忆。</strong></span>
## 十三、总结

Memory 是 Agent 从“单轮问答工具”走向“长期协作伙伴”的关键能力。

没有记忆，Agent 每次都只能从零开始。

有了记忆，Agent 才能理解用户、延续任务、复用经验，并在长期交互中变得更贴合实际需求。

但记忆系统不是简单地保存聊天记录。

真正好的 Memory 系统，应该具备：

- 短期上下文管理能力
- 长期信息存储能力
- 语义检索能力
- 记忆更新能力
- 遗忘与版本治理能力
- 防止记忆污染的机制

所以，Memory 的本质不是“让 Agent 记住一切”，而是：

在正确的时间，找出正确的历史信息，帮助 Agent 更好地完成当前任务。
