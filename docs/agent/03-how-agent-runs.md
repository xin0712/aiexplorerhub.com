:::tip 先看结论
Agent 的运行，不是“大模型一次性给答案”，而是围绕目标持续推进的执行闭环。

读完这篇你会明确三件事：

- Agent 从输入到输出的完整生命周期是什么
- LLM、Planner、Tools、Memory、Controller 在运行中如何协同
- 为什么 Agent 不是一次性问答，而是一个持续执行系统
:::

很多人第一次理解 Agent，会把它理解成一个“更聪明的 ChatGPT”。

但这个理解并不准确。

ChatGPT 这类问答模型，典型工作方式是：

用户输入问题 -> 模型生成答案

比如你问：

什么是 MCP？

模型回答：

MCP 是一种让模型连接外部工具和数据源的协议。

这个过程本质上是一次问答。

但 Agent 的目标不是只回答你，而是帮你完成一个任务。

比如你说：

帮我搭一个 MCP 示例项目。

这时候，Agent 不能只解释 MCP 是什么。

它需要继续做很多事情：

- 理解你到底要搭什么项目
- 判断项目需要哪些文件
- 生成目录结构
- 编写配置文件
- 调用工具创建文件
- 运行代码测试
- 发现报错后修正
- 最后告诉你项目是否跑通

这就不是“一问一答”了，而是一条完整的任务执行链路。

所以，Agent 的核心不是“回答能力”，而是“推进能力”。

它不是只负责说清楚，而是要把事情往前推进。

## 一个 Agent 的典型生命周期

一个 Agent 从接收到用户输入，到最终交付结果，通常会经历七个阶段：

- 解析用户目标
- 制定任务计划
- 选择并调用工具
- 观察执行结果
- 反思与纠错
- 交付最终结果
- 沉淀长期记忆

这七个阶段不是简单的直线流程。

真实运行时，它更像一个循环：

计划 -> 执行 -> 观察 -> 修正 -> 再执行

如果工具返回正常，Agent 继续推进。  
如果工具执行失败，Agent 尝试修正。  
如果信息不够，Agent 追加查询。  
如果结果有风险，Agent 暂停并请求人工确认。

这就是 Agent 和普通自动化脚本最大的区别。

普通自动化更像固定流水线：

如果 A 发生，就执行 B。

Agent 更像一个会根据情况调整动作的执行者：

目标是 X，我现在应该先做 A，还是先做 B？如果 A 失败了，我要不要换一种方式？

这就是 Agent 的运行闭环。

## 第一步：解析用户目标

Agent 运行的第一步，不是马上回答，而是先理解用户到底想完成什么。

用户输入通常是自然语言，而不是标准任务说明。

比如用户说：

帮我看一下昨天 847 的消耗和转化情况，判断有没有异常，并给一段内部同步话术。

这句话表面上只是一句话，但里面其实包含了很多任务信息：

- 任务类型：广告投放数据分析
- 分析对象：847
- 时间范围：昨天
- 关注指标：消耗、转化、ROI 等
- 输出要求：判断异常，并生成内部同步话术
- 潜在动作：查询数据、对比趋势、拆分维度、生成结论

这一步通常由 LLM 完成。

LLM 负责理解用户语言，把模糊请求转成结构化任务。

如果这一步理解错了，后面所有步骤都会跑偏。

比如用户说“847”，Agent 需要判断它到底是 campaign、adv_id、ad_group，还是内部项目编号。

如果上下文足够，它可以自动推断。

如果上下文不足，它就应该追问关键问题，而不是直接乱查。

所以，目标解析不是简单识别关键词，而是理解用户真正要完成的业务目标。

## 目标解析不是只看字面意思

Agent 理解目标时，不能只看用户说了什么，还要理解用户真正想达成什么。

比如用户说：

帮我看看今天 ROI 为什么掉了。

字面意思是“分析 ROI 下降原因”。

但真实任务可能包括：

- 今天 ROI 相比昨天掉了多少？
- 是整体下降，还是某几个广告组下降？
- 是消耗变高了，还是转化变少了？
- 是流量质量变差，还是出价策略出了问题？
- 是否有异常渠道、异常 bundle、异常素材？
- 最终应该怎么调整？

也就是说，用户真正要的不是一段“ROI 下降原因大全”。

用户真正要的是：

定位问题，并给出可执行建议。

这就是 Agent 和普通 Chatbot 的区别。

Chatbot 看到的是“问题”。  
Agent 看到的是“目标”。

## 第二步：制定任务计划

理解目标之后，Agent 不能马上执行。

它需要先制定计划。

这一步主要由 Planner / Controller 负责。

Planner 的作用，是把一个复杂目标拆成一组可执行步骤。

比如用户的目标是：

分析昨天 847 的消耗和转化异常。

Agent 可以拆成这样的计划：

- 确认 847 对应的业务对象
- 查询昨天整体消耗、曝光、点击、转化、ROI
- 对比前一天和近 7 日均值
- 按渠道拆分消耗变化
- 按广告组拆分转化变化
- 检查 CTR、CVR、winrate 是否异常
- 定位主要异常来源
- 生成原因判断
- 输出优化建议
- 生成内部同步话术

这就是任务规划。

没有 Planner 的 Agent，很容易想到哪做到哪。

查了一些数据，但没有完整链路。  
生成了一些结论，但缺少证据。  
看似回答很多，其实没有真正解决问题。

所以 Planner 的价值在于：

把模糊目标变成明确步骤。  
把复杂任务拆成可执行动作。  
把执行顺序安排清楚。

## 好的任务计划应该具备三个特点

一个好的 Agent 计划，至少要具备三个特点。

第一，步骤要可执行。

不能只是写：

分析数据。  
判断原因。  
输出结论。

这太粗了。

更好的计划应该是：

查询昨日消耗、曝光、点击、转化、ROI。  
对比前一日和近 7 日均值。  
按渠道、广告组、bundle 拆分变化。  
识别下降幅度最大的维度。

这样每一步都能真正执行。

第二，顺序要合理。

比如做广告分析，不能一上来就给建议。

应该先看整体，再看拆分。  
先定位异常，再判断原因。  
先验证数据，再输出结论。

第三，要有回退方案。

真实执行过程中，经常会出现问题：

SQL 报错。  
字段不存在。  
API 返回为空。  
数据口径不一致。  
权限不足。  
查询结果不完整。

如果没有回退方案，Agent 很容易卡住。

所以，一个更成熟的 Agent 计划里，通常还会包含：

如果查询失败，尝试换表。  
如果字段缺失，检查字段映射。  
如果结果为空，扩大时间范围。  
如果指标异常，追加拆分维度。

这就是 Agent 比普通自动化更灵活的地方。

## 第三步：选择并调用工具

计划制定完成之后，Agent 就要判断每一步应该用什么工具完成。

这一步体现的是 Tools 模块的价值。

因为很多真实任务，单靠 LLM 是做不了的。

比如：

查广告数据，需要 SQL 工具。  
读取文档，需要文件工具。  
搜索最新信息，需要搜索工具。  
运行代码，需要代码解释器。  
创建任务，需要项目管理工具。  
发送通知，需要飞书、Slack 或邮件工具。  
调用业务系统，需要 API。

所以，Agent 必须知道：

当前任务需要哪些工具？  
每个工具适合解决什么问题？  
调用工具需要传什么参数？  
工具返回结果后应该如何解释？  
如果工具失败，下一步怎么办？

比如广告分析任务中，Agent 可能会选择：

用 SQL 工具查询消耗和转化数据。  
用数据看板工具查看趋势。  
用知识库工具查历史优化策略。  
用飞书工具生成同步内容。

这一步决定了 Agent 能不能真正进入业务系统。

如果没有工具，Agent 只能“说应该怎么做”。

有了工具，Agent 才能“真的去做”。

## Tool Calling 让 AI 从“语言能力”变成“行动能力”

过去的大模型主要能力是语言能力。

它可以写。  
可以总结。  
可以解释。  
可以推理。

但它不能直接操作外部系统。

Tool Calling 出现之后，模型开始可以连接外部世界。

比如一个投放 Agent，可以调用：

SQL 查询工具  
广告平台 API  
反作弊系统  
素材库  
数据看板  
飞书通知  
知识库  
本地文件

这时 Agent 的能力边界就变了。

它不再只是“知道怎么做”，而是可以“真的去做”。

比如用户说：

帮我看一下昨天 847 的消耗为什么下降。

Agent 不是直接编原因，而是会执行：

查询昨天 847 消耗  
查询前一天消耗  
计算下降比例  
按渠道拆分  
按 ad group 拆分  
按素材拆分  
定位主要下降来源

这些动作都依赖工具。

所以，工具调用是 Agent 从 Chatbot 进化到执行系统的关键分水岭。

## 工具调用不是越多越好

这里要注意一点：

Agent 不是调用工具越多越好。

真正好的 Agent，应该知道：

什么时候需要调用工具。  
调用哪个工具最合适。  
传什么参数。  
如何理解工具返回结果。  
如果工具失败，下一步怎么办。

比如用户只是问：

什么是 Agent？

这时候不一定需要调用工具，直接解释就可以。

但用户问：

帮我查一下昨天广告 ROI 为什么下降。

这就必须调用数据工具。

所以，工具调用能力的核心不是“会调用”，而是“会选择”。

会选择工具，才是真正的执行智能。

## 第四步：观察执行结果

Agent 调用工具之后，不能只看“有没有返回结果”。

它还要看结果是否合理。

这一步叫 Observation，也就是观察。

比如 SQL 返回了数据，Agent 还要判断：

数据是否为空？  
数值是否异常？  
时间范围是否正确？  
指标口径是否匹配？  
是否需要进一步拆分？  
是否能支撑最终结论？

举个例子。

Agent 查询发现：

昨天 847 消耗下降了 35%。

这个结果本身还不够。

它还需要继续观察：

是所有渠道都下降，还是某个渠道下降？  
是曝光下降，还是点击率下降？  
是 winrate 下降，还是预算受限？  
是转化少了，还是流量质量变差？

如果它发现 Vungle 渠道下降明显，它就应该继续按渠道、广告组、bundle、人群包拆分。

这就是 Agent 的关键能力：

不是查到一个结果就结束，而是根据结果继续推进。

## 第五步：反思与纠错

Agent 的执行过程一定会遇到错误。

所以一个真正可用的 Agent，必须具备反思与纠错能力。

这一步通常叫 Reflection。

它的作用是让 Agent 在执行过程中不断检查：

当前结果是否符合预期？  
有没有遗漏关键步骤？  
工具返回结果是否可靠？  
结论是否有证据支撑？  
是否需要重新规划？  
是否需要人工确认？

比如 Agent 查询广告数据时发现：

昨天消耗为 0。

它不能马上说：

昨天没有消耗。

因为这可能是错的。

真实原因可能是：

查错表了。  
时间时区错了。  
数据还没同步。  
筛选条件写错了。  
账户 ID 识别错了。

所以它应该进行合理性校验：

换一个数据表验证。  
扩大时间范围验证。  
查询前一天是否有数据。  
检查时区是否正确。  
检查 847 是否对应正确对象。

这就是反思与纠错。

没有 Reflection 的 Agent，很容易“一本正经地给出错误答案”。

## Agent 的核心循环：行动、观察、再行动

Agent 的运行不是单线流程，而是循环流程。

可以理解为：

Plan -> Act -> Observe -> Reflect -> Act Again

计划。  
行动。  
观察。  
反思。  
再次行动。

举个例子。

用户让 Agent 分析消耗下降。

Agent 第一次查询发现：

整体消耗下降了 35%。

这还不够。

于是它继续观察：

是哪个渠道下降？  
是哪个广告组下降？  
是曝光少了，还是点击少了？  
是 winrate 降了，还是预算受限？

它发现：

主要是 Vungle 渠道曝光下降。

进一步查询发现：

某几个核心 ad group 的 winrate 明显下降。

再继续查，发现：

bid floor 上升，导致竞价竞争力下降。

最后它才能输出一个更可靠的结论：

昨日 847 消耗下降，主要不是素材点击率问题，而是 Vungle 渠道部分核心 ad group 的竞价竞争力下降，winrate 走低，导致曝光缩量。建议优先检查出价策略和核心流量包预算分配。

这个结论不是模型凭空生成的，而是经过多轮观察和验证得出的。

这就是 Agent 的价值。

## 第六步：交付最终结果

当 Agent 完成执行、观察和修正之后，才进入交付阶段。

交付不是把所有中间过程原样贴出来。

真正好的交付，应该是把复杂过程整理成用户能直接使用的结果。

比如广告分析任务的交付可以包括：

结论  
证据  
原因判断  
优化建议  
同步话术

例如：

结论：

昨天 847 消耗下降主要来自核心渠道曝光缩量，整体转化效率暂未明显恶化。

证据：

核心渠道消耗下降贡献了主要波动，部分 ad group 的 winrate 下降明显，但 CTR 和 CVR 相对稳定。

建议：

优先恢复核心流量包竞争力，保留 ROI 稳定的 CPE 组合，探索预算控制在 10%-20%。

内部同步话术：

目前 847 主要以已探索出的优质结果为主投放，包括一方人群定投和出单人群包重点投放。昨日消耗波动主要受核心流量曝光变化影响，整体转化效率暂未出现明显恶化。接下来会继续稳定核心 CPE 组合，同时保留少量 CPC 出价 ad group 用于补量，并将探索预算控制在合理范围内，避免低质流量放大。

这才是一个真正可用的 Agent 输出。

## 好的交付应该包含四个部分

一个成熟 Agent 的交付结果，通常应该包含四部分。

第一，结论。

先告诉用户最重要的判断是什么。

比如：

昨天消耗下降主要来自 Vungle 渠道曝光减少，不是整体转化率恶化。

第二，证据。

告诉用户这个判断是怎么来的。

比如：

Vungle 消耗下降贡献了整体下降的大部分；核心 ad group winrate 下降；CTR 和 CVR 没有明显恶化。

第三，建议。

告诉用户下一步怎么做。

比如：

优先恢复核心 ad group 出价竞争力，保留 ROI 稳定组合，探索预算控制在 10%-20%。

第四，话术。

把结论转成能直接同步的语言。

比如：

昨日整体消耗下滑主要受核心渠道曝光缩量影响，目前转化效率未出现明显恶化，接下来会优先恢复优质流量竞争力，同时控制探索预算，保证 ROI 稳定。

这就是一个可交付的 Agent 输出。

## 第七步：沉淀长期记忆

很多人会忽略最后一步：记忆沉淀。

但这是 Agent 从“临时工具”变成“长期助手”的关键。

Agent 执行完任务后，应该沉淀两类信息。

第一类是短期记忆。

比如当前任务里：

用户要分析 847。  
时间范围是昨天。  
已经查过整体数据。  
已经定位到核心渠道。  
下一步要看 ad group。

短期记忆用于保证当前任务不断线。

第二类是长期记忆。

比如：

用户常关注 ROI、event10、winrate、CTR、CVR。  
用户做广告分析时，喜欢按渠道、ad group、bundle、人群包拆分。  
用户希望输出内部同步话术。  
用户不希望 Agent 自动执行高风险投放动作。

长期记忆用于提升未来任务效率。

下次用户再说：

帮我看一下 847 今天有没有异常。

Agent 就不用从零开始问：

你关注哪些指标？  
你要什么输出格式？  
你希望按什么维度拆？

它可以直接沿用过去沉淀的偏好和规则。

这就是 Memory 的价值。

## 记忆不是简单保存聊天记录

很多人以为 Agent 的记忆就是“把聊天记录存下来”。

这其实不够。

真正有价值的记忆，不是原始对话，而是结构化经验。

比如原始对话是：

以后帮我分析广告数据时，重点看 ROI、event10、winrate，最后给我一段内部同步话术。

如果只是保存原文，价值有限。

更好的方式是沉淀成规则：

广告分析任务默认关注指标：ROI、event10、winrate、CTR、CVR。  
广告分析任务默认输出格式：结论、证据、建议、同步话术。  
高风险投放动作需要用户确认。

这才是可复用的记忆。

Agent 的记忆能力，不是为了“记住所有东西”，而是为了在未来任务中减少重复确认，提高执行质量。

## 从模块角度看 Agent 如何协同

结合上一篇讲的四个核心模块，Agent 的运行可以这样理解。

### LLM 负责理解和推理

LLM 是 Agent 的大脑。

它负责理解用户输入，识别任务意图，解释工具结果，生成最终表达。

但 LLM 只是大脑，不是完整系统。

如果只有 LLM，没有 Tools、Memory 和 Planner / Controller，它仍然只是一个问答模型。

### Planner / Controller 负责规划和调度

Planner 负责拆解任务。

Controller 负责控制流程。

它们决定：

下一步做什么。  
是否需要调用工具。  
是否需要重试。  
是否需要暂停等待用户确认。  
是否已经达到交付条件。

如果没有 Planner / Controller，Agent 就会缺少执行秩序。

它可能会回答很多内容，但很难稳定完成任务。

### Tools 负责真实执行

Tools 是 Agent 的手和脚。

它让 Agent 能够连接外部系统。

比如数据库、API、浏览器、文件系统、飞书、Notion、GitHub、广告平台。

没有 Tools，Agent 只能生成建议，无法完成动作。

### Memory 负责上下文和经验复用

Memory 让 Agent 不会每次都从零开始。

它保存用户偏好、业务规则、历史任务、成功经验和失败教训。

没有 Memory，Agent 很难形成长期协作能力。

## 用一个简单公式理解 Agent 运行

可以把 Agent 的运行理解成一个公式：

Agent = LLM 理解目标 + Planner 制定计划 + Tools 执行动作 + Controller 控制流程 + Memory 沉淀经验

其中：

LLM 解决“理解什么”。  
Planner 解决“怎么拆”。  
Tools 解决“怎么做”。  
Controller 解决“下一步做什么”。  
Memory 解决“以后如何做得更好”。

这几个模块协同起来，Agent 才能真正从“会回答”走向“会执行”。

## Agent 运行不是完全自动化

这里还要强调一点：

Agent 能执行任务，不代表它应该所有事情都自动做。

尤其在企业场景里，很多动作都存在风险。

比如：

删除数据  
发送客户邮件  
关闭广告组  
提高预算  
修改出价策略  
提交代码  
执行财务操作

这些动作不能一开始就完全交给 Agent 自动执行。

更合理的方式是：

低风险动作，Agent 可以自动完成。  
中风险动作，Agent 先生成建议。  
高风险动作，必须人工确认后执行。

比如投放 Agent 可以自动做：

查数据  
算指标  
找异常  
生成日报  
整理话术

但涉及这些动作时，应该让人确认：

是否关闭广告组  
是否提高预算  
是否调整出价  
是否拉黑 bundle  
是否发送正式客户说明

这就是生产级 Agent 必须具备的权限控制。

一个真正可用的 Agent，不只是能做事，还要知道哪些事不能擅自做。

## Agent 运行中最容易出问题的地方

Agent 看起来很强，但真正落地时，有几个地方最容易出问题。

### 目标理解错误

用户说得模糊，Agent 没有澄清，就可能执行错方向。

比如用户说：

帮我看一下数据。

这里的“数据”可能是投放数据、财务数据、用户行为数据，也可能是某个报表数据。

如果 Agent 直接执行，就容易跑偏。

### 计划拆得太粗

有些 Agent 只会写：

查询数据  
分析原因  
输出建议

这不是真正的计划。

真正可执行的计划应该具体到：

查哪个表  
看哪些指标  
按哪些维度拆  
和哪个时间段对比  
异常阈值是什么  
输出什么格式

计划越具体，执行越稳定。

### 工具调用不可靠

工具调用失败很常见。

比如 API 超时、字段变更、权限不足、返回为空。

Agent 必须能识别失败，并说明失败原因。

不能工具没查到，就假装已经查到了。

### 反思能力不足

有些 Agent 查到一个结果就直接下结论。

但很多结果需要二次验证。

比如消耗下降，不一定是转化问题。  
ROI 下降，不一定是流量质量问题。  
曝光下降，不一定是预算问题。

必须继续拆分，才能定位真实原因。

### 记忆污染

记忆不是越多越好。

错误记忆会让 Agent 越用越偏。

比如一次临时策略是“提高预算”，但 Agent 把它当成长期规则，以后每次都建议提高预算，这就很危险。

所以 Memory 必须可管理、可更新、可删除。

## 一个可生产化 Agent 的运行原则

如果要把 Agent 真正用到业务里，可以遵循几个原则。

第一，先从低风险、高频任务开始。

比如日报生成、数据分析、会议总结、需求整理、文档检查。

第二，所有工具调用都要有日志。

Agent 调用了什么工具、传了什么参数、返回了什么结果，都应该可追踪。

第三，高风险动作必须人工确认。

尤其是涉及钱、客户、权限、删除、发布、投放调整的动作。

第四，交付结果必须有证据。

不能只给结论，要说明结论来自哪些数据或执行结果。

第五，记忆要可控。

用户能知道 Agent 记住了什么，也能修改和删除错误记忆。

## 最后总结

Agent 是如何运行的？

一句话总结：

Agent 不是一次性回答问题，而是围绕用户目标，持续规划、执行、观察、修正和交付的闭环系统。

它的典型生命周期是：

解析用户目标 -> 制定任务计划 -> 调用工具执行 -> 观察执行结果 -> 反思与纠错 -> 交付最终结果 -> 沉淀长期记忆

上一篇我们讲 Agent 的核心组成：

LLM  
Memory  
Tools  
Planner / Controller

这一篇可以理解为这些模块的协同方式：

LLM 负责理解目标。  
Planner 负责拆解任务。  
Tools 负责执行动作。  
Controller 负责流程调度。  
Memory 负责经验沉淀。

只有这些模块形成闭环，Agent 才不只是一个会聊天的模型，而是一个能进入真实工作流的执行系统。

这就是 Agent 的运行方式。
