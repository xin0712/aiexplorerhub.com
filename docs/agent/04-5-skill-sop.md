本章介绍 Skill 如何把经验、流程、规则和工具使用方式封装成可复用能力，让 Agent 从通用聊天助手变成专业任务执行者。

这篇你会理解三件事：

- Skill 到底是什么，为什么它是 Agent 专业化的关键中间层
- Skill 与 Prompt、Tool Calling 的区别，以及各自负责什么
- Skill 如何与 Memory、Tool Calling、MCP 协同，把任务从“能回答”升级为“能执行”

上一篇我们讲了 Memory。

Memory 解决的是 Agent 的“记忆问题”。

它让 Agent 不只是一次性回答问题，而是可以记住用户偏好、历史任务、项目背景和长期上下文。

但光有记忆还不够。

因为一个 Agent 即使记住了很多信息，也不代表它真的“会做事”。

比如你告诉一个 Agent：

我是广告投放产品经理，经常要分析 ROI、CTR、CVR、作弊率，还要写飞书播报。

它记住这些信息以后，下次确实可以更懂你的背景。

但如果它没有一套稳定的方法，它每次分析广告数据时可能都会不一样：

有时看 ROI。  
有时看 CTR。  
有时忘了看 CVR。  
有时忽略作弊率。  
有时只给结论，不给原因。

这就说明，Memory 只解决了“记住什么”，但没有解决“怎么做”。

而 Skill 解决的正是这个问题。

简单说：

Skill 是 Agent 针对某一类任务沉淀下来的能力包和操作 SOP。

它让 Agent 不只是临场发挥，而是可以按照一套固定的方法、流程和判断标准完成任务。

## 一、什么是 Skill？

Skill 可以理解为 Agent 的“专项能力”。

它不是单个工具，也不是一句 Prompt，而是一套完成某类任务的方法封装。

比如：

广告数据分析 Skill  
SQL 排查 Skill  
股票盘前分析 Skill  
简历优化 Skill  
小红书图文生成 Skill  
课程讲稿改写 Skill  
需求分析 Skill

每一个 Skill 都对应一种相对稳定的任务类型。

比如“广告数据分析 Skill”，它可能包含：

1. 需要先确认时间范围、业务类型、adv_id、渠道等条件  
2. 再查询消耗、收入、ROI、转化数  
3. 再拆解 imp、click、lp_click、purchase 漏斗  
4. 再判断 CTR、CVR、winrate 是否异常  
5. 再结合 fraud 指标判断是否存在假量  
6. 最后输出问题、原因和优化建议

这就是一个 Skill。

它不是告诉 Agent “调用某个工具”这么简单，而是告诉 Agent：

面对这一类任务，应该按照什么逻辑去分析、判断和输出。

所以 Skill 更像是人的专业经验。

一个新手看广告数据，可能只会看收入涨没涨。

一个有经验的投放分析师，会看 ROI、成本、转化漏斗、渠道、素材、出价、作弊率、预算消耗节奏。

Skill 要做的，就是把这些经验封装给 Agent。

## 二、为什么 Agent 需要 Skill？

因为大模型虽然能力很强，但它天然有一个问题：

它每次回答都可能不稳定。

同样一个问题，今天它可能这么分析，明天可能换一个角度，后天又漏掉关键步骤。

对于普通聊天，这个问题不大。

但对于严肃任务，这就会带来很大风险。

比如你让 Agent 帮你做广告投放复盘，如果它漏看作弊率，可能会把低质量流量误判成“素材问题”。

你让 Agent 帮你做 SQL 排查，如果它没有先判断是语法错误、资源错误还是权限错误，就可能给出错误方案。

你让 Agent 帮你判断股票买点，如果它只看价格，不看趋势、成交量、支撑位、压力位、市场情绪，就容易误导决策。

所以 Agent 需要 Skill 的原因是：

把不稳定的模型能力，变成相对稳定的任务执行能力。

Skill 的价值不是让模型“知道更多”，而是让模型“做得更稳”。

## 三、Skill 和 Prompt 有什么区别？

很多人会把 Skill 和 Prompt 混在一起。

它们确实有关系，但不是同一个东西。

Prompt 更像是一段临时指令。

比如：

请你作为一名广告投放专家，帮我分析下面的数据。

这是一句 Prompt。

它可以引导模型进入某个角色，但它通常是临时的、一次性的。

而 Skill 更像是一套长期可复用的能力包。

它不仅包含角色，还包含：

- 任务适用场景
- 执行步骤
- 输入要求
- 工具使用方式
- 判断规则
- 异常处理
- 输出格式
- 质量标准

所以 Prompt 更偏“告诉模型这次怎么回答”。

Skill 更偏“沉淀一类任务的标准做法”。

可以这样理解：

| 对比项 | Prompt | Skill |
| --- | --- | --- |
| 本质 | 临时指令 | 可复用能力包 |
| 作用 | 引导一次回答 | 稳定完成一类任务 |
| 内容 | 通常较短 | 通常包含流程、规则、工具、格式 |
| 生命周期 | 一次对话内有效 | 可以长期复用 |
| 目标 | 让模型回答得更像某个角色 | 让 Agent 稳定执行某类任务 |

比如你每次都写：

请你先看 ROI，再看 CTR、CVR、作弊率，最后输出建议。

这就是在反复写 Prompt。

但如果你把它沉淀成“广告投放分析 Skill”，以后 Agent 只要识别到这是广告分析任务，就可以自动套用这套流程。

这就是 Skill 的价值。

## 四、Skill 和 Tool Calling 有什么区别？

Skill 和 Tool Calling 也容易混淆。

Tool Calling 是工具调用能力。

它解决的是：

Agent 怎么调用外部工具。

比如：

调用 SQL 工具  
调用搜索工具  
调用天气工具  
调用飞书发送工具  
调用文件读取工具  
调用图片生成工具

而 Skill 解决的是：

Agent 为什么调用这些工具、先调用哪个、拿到结果后怎么判断。

举个例子。

如果用户说：

帮我分析昨天 Nutra 的 ROI，找出 ROI 低于 0.8 的渠道，并生成飞书播报。

Tool Calling 负责执行这些动作：

调用 SQL 查询数据  
调用反作弊工具查询 fraud 指标  
调用飞书工具发送消息

但 Skill 会告诉 Agent：

1. 先按渠道聚合 spend、revenue、conversion、ROI  
2. 筛选 ROI < 0.8 的渠道  
3. 对异常渠道继续拆 CTR、CVR、winrate  
4. 如果 CTR 高但 CVR 低，要怀疑流量质量或落地页问题  
5. 如果 click_cheat_ratio 高，要标记可能存在作弊流量  
6. 最后按照飞书播报格式输出

也就是说：

Tool Calling = 能调用工具  
Skill = 知道怎么组合工具完成任务

没有 Tool Calling，Agent 只能建议，不能执行。

没有 Skill，Agent 即使能调用工具，也可能乱查、漏查、误判。

## 五、一个 Skill 通常包含什么？

一个完整的 Skill，通常可以包含八个部分。

### 1. 任务定义

说明这个 Skill 用来解决什么问题。

比如：

广告投放分析 Skill 用于分析广告业务的消耗、收入、ROI、转化漏斗、渠道质量和作弊风险，并输出优化建议。

### 2. 触发场景

说明什么情况下应该使用这个 Skill。

比如用户说：

分析投放效果  
看一下 ROI  
查一下昨天消耗  
找出低质渠道  
生成飞书日报  
复盘广告数据

这些都可以触发广告分析 Skill。

### 3. 输入要求

说明需要哪些信息。

比如：

时间范围  
业务类型  
adv_id  
渠道  
国家  
ad_format  
统计维度

如果用户没有提供关键字段，Agent 就要追问。

比如用户只说：

帮我看一下昨天的数据。

Agent 就应该追问：

你要看哪个业务、哪个 adv_id，还是整体数据？

### 4. 执行步骤

这是 Skill 的核心。

它规定 Agent 应该按什么顺序做事。

比如：

1. 确认分析目标
2. 查询核心指标
3. 拆解转化漏斗
4. 识别异常点
5. 定位原因
6. 输出建议

### 5. 工具使用方式

说明这个 Skill 可能需要调用哪些工具。

比如：

SQL 查询工具  
数据看板工具  
反作弊查询工具  
飞书发送工具  
搜索工具  
文件读取工具

Skill 不一定自己执行工具，但它要告诉 Agent 在什么时候应该使用什么工具。

### 6. 判断规则

这是 Skill 的专业性来源。

比如广告分析里可以有这些规则：

ROI 下降，要拆 spend、revenue、conversion  
CTR 下降，优先看素材吸引力和流量匹配  
CVR 下降，优先看落地页、转化链路和流量质量  
winrate 下降，优先看 bid_price、bid_floor、竞争强度  
作弊率升高，要结合 IP、device_id、click_cheat_ratio、postback 质量判断

这些规则就是经验。

没有这些规则，Agent 只是机械地查数据。

有了这些规则，Agent 才能像一个懂业务的人一样分析问题。

### 7. 异常处理

真实任务里经常会出错。

比如：

SQL 执行失败  
接口超时  
数据为空  
权限不足  
参数缺失  
字段不存在  
结果明显异常

Skill 应该规定遇到这些情况怎么处理。

比如 SQL 报错：

`Exceed concurrency limit`

Agent 应该知道这不是业务数据问题，而是执行资源或并发限制问题。

它应该建议：

稍后重试  
缩小查询范围  
减少扫描字段  
增加 dt 分区条件  
避免 SELECT *

### 8. 输出格式

Skill 还应该规定最终怎么表达。

比如广告分析结果可以固定输出为：

结论  
核心数据  
异常点  
原因判断  
优化建议  
下一步动作

这样用户每次收到的结果都是稳定的，而不是一会儿长篇大论，一会儿只给一句话。

## 六、Skill 的基本结构示例

一个 Skill 可以用类似下面的结构来描述：

Skill 名称：广告投放分析 Skill  
目标：  
用于分析广告投放效果，识别 ROI、CTR、CVR、winrate、作弊率等关键指标异常，并输出优化建议。  
触发场景：  
当用户要求分析广告效果、ROI、消耗、渠道质量、转化漏斗、异常原因、飞书播报时触发。  
输入参数：

- 时间范围
- 业务类型
- adv_id
- 渠道
- 国家
- ad_format
- 分析维度

执行步骤：

1. 确认分析范围
2. 查询 spend、revenue、conversion、ROI
3. 拆解 imp、click、lp_click、purchase 漏斗
4. 分析 CTR、CVR、winrate
5. 检查 fraud 指标
6. 定位主要问题
7. 输出建议和播报内容

可用工具：

- SQL 查询工具
- 反作弊查询工具
- 飞书消息工具

判断规则：

- ROI 低于目标值，优先拆收入和成本
- CTR 低，优先判断素材和流量匹配问题
- CVR 低，优先判断落地页和流量质量问题
- 作弊率高，优先建议限制低质渠道、bundle、tagid
- winrate 低，优先判断 bid_floor 和出价竞争问题

输出格式：

1. 总体结论
2. 核心数据
3. 异常原因
4. 优化建议
5. 下一步动作

这就是一个比较完整的 Skill。

## 七、广告投放分析 Skill 示例

假设用户说：

帮我分析昨天 SOI 的投放效果，重点看 ROI 低的渠道。

如果 Agent 有广告投放分析 Skill，它不会直接泛泛回答：

建议优化素材，提高转化率。

它会按照 Skill 的流程执行：

第一步，确认分析目标：

分析对象：SOI  
时间范围：昨天  
重点指标：ROI  
分析维度：渠道

第二步，查询核心指标：

spend  
revenue  
conversion  
ROI  
CTR  
CVR  
winrate  
fraud ratio

第三步，识别异常：

哪些渠道 ROI < 目标值  
哪些渠道消耗高但转化少  
哪些渠道点击多但 purchase 少  
哪些渠道作弊率异常

第四步，判断原因：

如果 CTR 低，可能是素材吸引力不足  
如果 CTR 高但 CVR 低，可能是流量质量或落地页问题  
如果 winrate 低，可能是出价竞争力不足  
如果 fraud 指标高，可能是低质流量或作弊流量

第五步，输出建议：

昨日 SOI 整体 ROI 偏低，主要由 channel A 和 channel B 拉低。  
channel A 的问题是 CTR 正常但 CVR 明显偏低，说明点击质量较差，建议优先排查该渠道下的 bundle 和 tagid，暂停低转化流量。  
channel B 的问题是 winrate 下降，同时 bid_floor 抬高，说明竞争成本上升，建议降低预算或调整出价策略。  
下一步建议：

1. 保留 ROI > 1 的渠道继续放量；
2. 暂停 ROI < 0.6 且 CVR 低的低质渠道；
3. 对 CTR 高但 CVR 低的流量补充 fraud 排查；
4. 每日跟踪 purchase、event10 和 click_cheat_ratio。

这就比普通回答更像一个真正的广告分析 Agent。

## 八、股票盘前分析 Skill 示例

再看一个投资场景。

用户说：

帮我判断今天 SOXL 更大概率涨还是跌。

如果没有 Skill，模型可能只会说：

SOXL 是三倍做多半导体 ETF，波动较大，请注意风险。

这没有太大帮助。

但如果有“股票盘前分析 Skill”，它会按固定框架判断：

1. 看纳指期货方向
2. 看半导体板块盘前表现
3. 看 NVDA、AMD、AVGO、TSM 等核心成分股
4. 看 SOXL / SOXS 的盘前相对强弱
5. 看重大新闻、财报、宏观事件
6. 看技术位：支撑、压力、均线、成交量
7. 给出方向判断、置信度和关键价位

它最终输出就会更稳定：

今日 SOXL 判断：偏多，但置信度中等。  
原因：

1. 纳指期货盘前走强，科技股风险偏好回升；
2. 半导体核心股盘前多数上涨，NVDA 和 AMD 对 SOXL 有正向支撑；
3. SOXS 盘前走弱，反向验证 SOXL 短线强度；
4. 但 SOXL 接近上方压力位，如果开盘后不能放量突破，容易冲高回落。

操作上不建议直接满仓追高，可以等开盘后 15 分钟确认方向。  
如果站稳关键压力位，再考虑右侧跟随；  
如果冲高无量并跌回均线下方，则优先观望。

这里真正重要的不是结论，而是背后的分析流程稳定。

这就是 Skill 的作用。

## 九、Skill 如何和 Memory 配合？

Memory 和 Skill 是互补关系。

Memory 负责记住用户的长期信息。

Skill 负责完成任务的方法。

比如用户长期偏好是：

用户做短线交易  
关注 SOXL、SOXS、TQQQ、NVDA  
不喜欢模棱两可的判断  
希望给出明确方向、置信度、支撑位、压力位  
单笔交易不想 all in

这些属于 Memory。

而股票分析 Skill 会负责：

分析市场趋势  
判断盘前强弱  
拆解技术指标  
输出买卖策略  
提示风险

当两者结合起来，Agent 的回答就会更贴合用户。

比如同样是分析 SOXL，普通 Skill 可能只输出市场分析。

但结合 Memory 后，Agent 会知道用户更关心：

今天要不要买  
盘前能不能进  
低吸点在哪里  
止损止盈怎么设  
仓位不能太重

所以可以总结为：

Memory 让 Agent 更懂用户  
Skill 让 Agent 更会做事

## 十、Skill 如何和 Tool Calling 配合？

Skill 往往会指导 Tool Calling。

比如一个广告分析 Skill 会告诉 Agent：

要先调用 SQL 工具查核心数据  
如果 ROI 异常，再查渠道维度  
如果 CVR 异常，再查漏斗数据  
如果作弊率异常，再查反作弊系统  
最后调用飞书工具发送结果

也就是说，Skill 决定工具调用的策略。

Tool Calling 负责真正执行。

二者关系可以这样理解：

Skill：我知道这件事应该怎么做  
Tool Calling：我现在去调用工具把它做出来

如果没有 Skill，Agent 可能不知道该查什么。

如果没有 Tool Calling，Agent 即使知道方法，也无法拿到真实数据。

所以一个能落地的 Agent，通常需要两者配合。

## 十一、Skill 如何和 MCP 配合？

MCP 解决的是工具标准化接入问题。

Skill 不关心工具底层是怎么连接的。

它只关心：

我完成任务时，需要哪些能力？  
这些能力有没有可用工具？  
工具返回结果后，我应该怎么判断？

而 MCP 负责让这些工具以统一方式暴露给 Agent。

比如广告分析 Agent 可能需要：

数据库 MCP Server  
飞书 MCP Server  
反作弊 MCP Server  
文件系统 MCP Server  
浏览器 MCP Server

Skill 会定义：

分析广告数据时，需要查询数据库；  
生成日报后，需要发送飞书；  
发现作弊异常时，需要查询反作弊系统。

MCP 则提供这些工具的标准连接方式。

所以三者关系是：

Skill 决定怎么做  
Tool Calling 负责调用  
MCP 负责标准化接入工具

## 十二、为什么 Skill 是 Agent 专业化的关键？

很多人理解 Agent 时，会把重点放在工具上。

比如：

Agent 能不能查网页？  
能不能写代码？  
能不能发邮件？  
能不能连数据库？

这些当然重要。

但真正决定 Agent 专业程度的，不只是它能调用多少工具，而是它是否知道：

什么时候调用工具  
为什么调用这个工具  
调用后怎么判断结果  
下一步该做什么  
最后怎么输出给用户

这就是 Skill 的价值。

一个没有 Skill 的 Agent，就像一个会使用各种工具的新手。

它有锤子、螺丝刀、电钻，但不知道什么时候该用哪个，也不知道怎么把它们组合起来完成一个项目。

一个有 Skill 的 Agent，更像一个熟练工人。

它不只是有工具，而是有方法、有流程、有经验。

所以 Skill 是 Agent 从“工具调用器”升级成“专业执行者”的关键。

## 十三、Skill 的真正价值

Skill 的真正价值可以总结为四点。

1. 让 Agent 更稳定

同类任务每次都按统一流程处理，不会随机发挥。

2. 让经验可以复用

人的经验可以被沉淀成 Skill，让 Agent 长期使用。

3. 让工具调用更有策略

不是为了调用工具而调用工具，而是围绕任务目标调用工具。

4. 让 Agent 更容易专业化

不同领域可以沉淀不同 Skill。

比如：

广告投放 Skill  
数据分析 Skill  
金融交易 Skill  
产品需求 Skill  
内容创作 Skill  
代码调试 Skill  
客户沟通 Skill

每多一个 Skill，Agent 就多一种稳定完成任务的能力。

## 十四、总结

Memory 解决的是：

Agent 记住什么。

Skill 解决的是：

Agent 会做什么，以及按什么方法做。

Tool Calling 解决的是：

Agent 怎么调用工具执行。

MCP 解决的是：

工具怎么标准化接入 Agent。

所以在 Agent 架构里，Skill 是一个非常关键的中间层。

它连接了记忆和工具。

没有 Memory，Agent 不懂用户背景。

没有 Skill，Agent 没有稳定方法。

没有 Tool Calling，Agent 无法执行动作。

没有 MCP，工具接入会越来越混乱。

如果说大模型是 Agent 的大脑，Memory 是 Agent 的记忆，那么 Skill 就是 Agent 的专业能力。

它让 Agent 不只是“知道很多”，而是“会按正确的方法做事”。

这也是 Agent 从通用聊天助手走向专业任务执行系统的关键一步。
