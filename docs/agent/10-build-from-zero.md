:::tip 先看结论
从 0 开始构建 Agent，第一目标不是做复杂系统，而是先跑通最小闭环：**理解任务 -> 调用工具 -> 观察结果 -> 输出结论**。

这篇你会掌握三件事：

- 如何定义一个“可落地”的最小可用 Agent
- 从任务边界到工具、Memory、Controller 的完整搭建路径
- 第一版该做什么、不该做什么，避免一开始就失控
:::

前面我们讲了 Agent 的核心组成、Tool Calling、MCP、Runtime、Multi-Agent 和企业级 Agent 架构。

这些内容偏理论。

这一篇我们进入实战：

如果从 0 开始，怎么搭建一个最小可用的 Agent？

这里的目标不是一上来做一个非常复杂的企业级 Agent，而是先构建一个能跑通闭环的 Agent。

一个最小可用 Agent，至少要具备四个能力：

1. 能理解任务
2. 能调用工具
3. 能记住过程
4. 能根据结果继续行动

也就是说，它不是简单问答机器人，而是可以围绕一个目标持续执行的小型智能系统。

## 一、先明确：我们要做的不是聊天机器人

普通聊天机器人是：

用户问一句  
模型答一句  
对话结束

Agent 更像是：

用户给一个目标  
Agent 拆解任务  
Agent 调用工具  
Agent 观察结果  
Agent 修正下一步  
Agent 输出最终结果

区别在于：

聊天机器人主要负责回答；Agent 负责完成任务。

例如用户说：

帮我分析昨天 SOI 业务 ROI 为什么下降。

普通聊天机器人可能会直接说：

可能是流量质量下降、转化率下降、预算变化导致的。

但 Agent 应该这样做：

1. 查询昨天 SOI 的消耗、收入、ROI
2. 对比前一天和近 7 日均值
3. 拆分 channel、adv_id、bundle、tagid
4. 检查 CTR、CVR、purchase、作弊率
5. 找出异常维度
6. 输出原因和建议

所以，从 0 开始构建 Agent，第一步不是写 Prompt，而是明确：

这个 Agent 到底要帮用户完成什么任务？

## 二、第一步：明确任务边界

不要一开始就做“万能 Agent”。

万能 Agent 听起来很强，但实际很难落地。

因为它会遇到几个问题：

任务范围太大  
工具太多  
权限难控  
结果难评估  
出错难定位

正确做法是先做一个窄场景 Agent。

比如：

日报生成 Agent  
SQL 分析 Agent  
投放诊断 Agent  
文案改写 Agent  
股票盘前分析 Agent  
客服知识库 Agent  
简历优化 Agent

任务越清晰，Agent 越容易做好。

### 1. 用一句话定义 Agent

构建 Agent 前，先用一句话定义它：

这个 Agent 帮谁，在什么场景下，完成什么任务。

例如：

这个 Agent 帮广告投放人员，自动分析昨日投放数据异常，并输出优化建议。

这句话里包含三个关键点：

用户是谁：广告投放人员  
场景是什么：昨日投放数据异常分析  
结果是什么：输出优化建议

这样边界就清楚了。

### 2. 明确它不做什么

这一步非常重要。

很多 Agent 做不好，不是因为能力不够，而是边界太模糊。

例如投放诊断 Agent，可以先规定：

它可以查询数据  
它可以分析原因  
它可以给出建议  
它可以生成日报

但第一版不要让它：

自动暂停广告组  
自动修改预算  
自动发送客户邮件  
自动调整出价

也就是：

第一版先做辅助决策，不做自动执行。

这会让系统安全很多。

## 三、第二步：设计 Agent 的输入和输出

Agent 不是随便聊，它应该有相对稳定的输入和输出结构。

### 1. 输入结构

用户可能会自然语言输入：

帮我看下昨天 SOI 为什么掉了。

但系统内部最好转成结构化任务：

```json
{
  "task_type": "performance_diagnosis",
  "business_type": "SOI",
  "date_range": "yesterday",
  "metrics": ["cost", "revenue", "roi", "ctr", "cvr", "purchase"],
  "output_type": "analysis_report"
}
```

这样 Agent 才能稳定执行。

因为模型理解自然语言可能会波动，但结构化任务更稳定。

### 2. 输出结构

输出也不要完全自由发挥。

可以固定成：

1. 结论
2. 关键数据
3. 异常原因
4. 建议动作
5. 后续观察指标

例如：

结论：  
昨天 SOI ROI 下滑主要来自 vungle 渠道 CVR 下降。  
关键数据：  
整体 ROI 从 1.05 降至 0.78，消耗上升 18%，purchase 下降 22%。  
异常原因：  
vungle 渠道消耗占比提升，但后端转化未同步增长。  
建议动作：  
1. 降低低 ROI bundle 出价 10%-15%  
2. 暂停连续两天无 purchase 的 tagid  
3. 保留 ROI > 1 的广告组继续扩量  
后续观察：  
明天重点观察 CVR、purchase、click_cheat_ratio。

固定输出结构的好处是：

用户更容易看懂  
结果更容易评估  
后续可以接入日报或飞书模板

## 四、第三步：设计 Prompt

Prompt 是 Agent 的行为说明书。

一个最小可用 Agent 的 Prompt，至少要包含五部分：

角色  
目标  
边界  
流程  
输出格式

可以写成这样：

```text
你是一个广告投放数据分析 Agent。
你的目标是帮助投放人员分析广告业务表现异常的原因，并输出可执行的优化建议。
你可以分析消耗、收入、ROI、CTR、CVR、purchase、渠道、广告主、bundle、tagid、作弊率等指标。
你不能直接执行暂停广告组、修改预算、调整出价等高风险操作，只能生成建议清单。
分析流程：
1. 先判断整体指标是否异常
2. 再和前一天、近 7 日均值对比
3. 然后按 channel、adv_id、bundle、tagid 拆分
4. 判断是流量质量问题、转化问题、出价问题还是数据回传问题
5. 最后输出结论和建议
输出格式：
1. 核心结论
2. 关键数据
3. 原因分析
4. 建议动作
5. 后续观察指标
```

这就是一个基础 Agent Prompt。

## 五、第四步：设计工具接口

Agent 真正有价值的地方，是能调用工具。

如果没有工具，它只能根据已有知识回答，很容易变成“猜”。

对投放分析 Agent 来说，第一版可以设计 3 到 5 个工具。

例如：

`query_overview_metrics`  
`query_channel_breakdown`  
`query_adv_breakdown`  
`query_bundle_breakdown`  
`query_fraud_metrics`

### 1. 工具不要太万能

不要一开始给 Agent 一个：

`execute_sql(sql)`

这会非常危险，也不稳定。

更推荐使用结构化工具：

`query_overview_metrics(date, business_type)`  
`query_channel_breakdown(date, business_type)`  
`query_bundle_breakdown(date, business_type, channel)`

这样 Agent 不需要自己随便拼复杂 SQL，而是调用你定义好的能力。

### 2. 工具要有明确输入输出

例如：

工具名称：`query_overview_metrics`  
作用：查询某业务在指定日期的核心投放指标。  
输入：

- `date`
- `business_type`

输出：

- `cost`
- `revenue`
- `roi`
- `impression`
- `click`
- `ctr`
- `conversion`
- `cvr`
- `purchase`

对 Agent 来说，工具描述越清楚，调用越稳定。

### 3. 第一版工具越少越好

很多人一开始就想接入几十个工具。

这会导致 Agent 不知道用哪个。

第一版只需要最关键的工具。

比如投放诊断 Agent：

第一个工具：查整体数据  
第二个工具：按渠道拆分  
第三个工具：按广告组拆分  
第四个工具：查作弊指标  
第五个工具：生成报告

够了。

先跑通闭环，再慢慢增加工具。

## 六、第五步：加入 Memory 和记忆机制

Agent 要完成复杂任务，不能每一步都从零开始。

它需要记住一些内容。

Memory 可以分成三类。

### 1. 短期记忆

短期记忆就是当前任务中的上下文。

例如：

用户要分析 SOI  
时间是昨天  
当前已经查过整体数据  
发现 ROI 下降  
下一步要查渠道拆分

这类记忆通常保存在一次任务的运行状态里。

### 2. 长期记忆

长期记忆是跨任务复用的信息。

例如：

SOI 常用 adv_id 有哪些  
Nutra 常用 adv_id 有哪些  
用户偏好输出简洁版日报  
ROI 小于 0.8 需要重点预警  
vungle 渠道需要重点看 CVR

这些信息可以存在数据库、配置文件、知识库或向量库里。

### 3. 操作日志

操作日志也是一种重要记忆。

它记录 Agent 做过什么：

调用了哪个工具  
传了什么参数  
工具返回了什么  
模型做了什么判断  
最终输出了什么

日志的价值是：

方便排查错误  
方便评测质量  
方便复盘优化  
方便审计追踪

所以，从 0 开始构建 Agent 时，日志不要等后面再加。

第一版就要有。

## 七、第六步：加入 Controller 执行循环

一个 Agent 不是只调用一次模型。

更常见的是一个循环：

理解任务  
↓  
制定计划  
↓  
选择工具  
↓  
执行工具  
↓  
观察结果  
↓  
判断是否继续  
↓  
输出最终结果

这就是最基础的 Agent Loop。

可以抽象成：

```python
while task_not_finished:
    think()
    choose_action()
    execute_tool()
    observe_result()
    update_state()
final_answer()
```

这个循环里，Controller 的职责是：

控制当前执行到哪一步  
决定下一步调用哪个工具  
判断工具结果是否有效  
判断是否需要重试  
判断是否应该停止

没有 Controller，Agent 就容易失控。

比如一直调用工具，或者工具失败后继续胡编。

## 八、第七步：加入失败处理

最小可用 Agent 也必须有失败处理。

因为真实环境里一定会失败。

常见失败包括：

参数缺失  
工具报错  
数据为空  
接口超时  
权限不足  
模型输出格式错误  
分析结果不确定

第一版可以设计简单的失败规则。

例如：

如果参数缺失，就追问用户  
如果工具超时，最多重试 2 次  
如果数据为空，提示数据未产出  
如果权限不足，提示无法访问  
如果输出格式错误，重新生成一次  
如果高风险动作，必须请求确认

这比让 Agent 自由发挥安全很多。

### 示例：数据为空时

错误做法：

昨天 SOI 下滑可能是因为渠道质量不好。

正确做法：

当前未查询到昨天 SOI 的完整数据，可能是数据尚未产出或业务参数不完整。建议先确认日期口径和业务范围，暂不输出确定性结论。

Agent 最重要的能力之一，不是“永远给答案”，而是：

知道什么时候不能给确定答案。

## 九、第八步：加入评测机制

Agent 上线后，必须评估它到底做得好不好。

最小可用 Agent 可以先评估几个指标：

任务是否完成  
工具调用是否正确  
输出格式是否符合要求  
结论是否有数据支撑  
建议是否可执行  
是否出现胡编  
是否需要人工返工

比如投放分析 Agent，可以用这些标准评测：

是否正确识别业务类型  
是否正确识别时间范围  
是否查询了核心指标  
是否做了环比或近 7 日对比  
是否找到了异常贡献维度  
是否区分 CTR 问题和 CVR 问题  
是否输出了明确建议

评测的价值是：

让 Agent 从“感觉还行”，变成“可持续优化”。

没有评测，就不知道该优化 Prompt、工具，还是业务流程。

## 十、一个最小可用 Agent 的完整流程

以“投放诊断 Agent”为例，第一版可以这样设计。

### 用户输入

帮我分析昨天 SOI ROI 为什么下降。

### 第一步：任务理解

系统解析为：

```json
{
  "task_type": "performance_diagnosis",
  "business_type": "SOI",
  "date_range": "yesterday",
  "risk_level": "low",
  "need_human_confirm": false
}
```

### 第二步：生成计划

1. 查询昨天 SOI 整体指标
2. 查询前一天和近 7 日均值
3. 查询渠道拆分
4. 查询 adv_id 拆分
5. 查询 bundle / tagid 异常
6. 查询作弊率
7. 汇总原因和建议

### 第三步：调用工具

```text
query_overview_metrics(date="yesterday", business_type="SOI")
query_overview_metrics(date="previous_day", business_type="SOI")
query_channel_breakdown(date="yesterday", business_type="SOI")
query_adv_breakdown(date="yesterday", business_type="SOI")
query_fraud_metrics(date="yesterday", business_type="SOI")
```

### 第四步：观察结果

Agent 拿到工具返回的数据后，不是直接贴数据，而是分析：

整体 ROI 是否下降  
消耗是否异常上升  
收入是否异常下降  
CTR 是否下降  
CVR 是否下降  
purchase 是否减少  
哪个渠道贡献了主要下滑  
哪个 adv_id 问题最大  
作弊率是否异常升高

### 第五步：生成结论

最终输出：

核心结论：  
昨天 SOI ROI 下滑主要来自 vungle 渠道 CVR 下降，而不是点击率问题。  
关键数据：  
整体 ROI 从 1.02 降至 0.76。  
CTR 基本稳定，但 CVR 下降 28%。  
vungle 渠道消耗占比从 25% 提升到 41%，但 purchase 下降 19%。  
原因分析：

1. vungle 新增流量质量偏弱
2. 部分 bundle 点击正常但后端转化低
3. 扩量后进入更宽流量池，导致 ROI 被拉低

建议动作：

1. 暂停连续两天无 purchase 的 bundle
2. 对低 ROI tagid 降价 10%-15%
3. 保留 ROI > 1 的 adv_id 继续扩量
4. 明天重点监控 CVR、purchase、click_cheat_ratio

这就是一个完整 Agent 闭环。

可以用下面这张图理解：

```text

│          用户输入             │
└──────────────┬───────────────┘
               ↓
┌──────────────────────────────┐
│        任务理解模块           │
│  意图识别 / 参数抽取 / 分类    │
└──────────────┬───────────────┘
               ↓
┌──────────────────────────────┐
│          Planner              │
│       拆解任务步骤             │
└──────────────┬───────────────┘
               ↓
┌──────────────────────────────┐
│         Controller            │
│   调度工具 / 控制状态 / 重试    │
└──────────────┬───────────────┘
               ↓
┌──────────────────────────────┐
│          Tool 层              │
│   SQL / API / 搜索 / 报表      │
└──────────────┬───────────────┘
               ↓
┌──────────────────────────────┐
│          Runtime              │
│     执行工具并返回结果          │
└──────────────┬───────────────┘
               ↓
┌──────────────────────────────┐
│       Memory / Logs           │

               ↓
┌──────────────────────────────┐
│          最终输出              │
└──────────────────────────────┘
```

## 十二、从 0 到 1 的推荐开发顺序

不要一开始就写复杂系统。

推荐按照这个顺序做：

第一步：写清楚 Agent 的任务边界  
第二步：设计输入输出格式  
第三步：写基础 Prompt  
第四步：接入 1-2 个核心工具  
第五步：加入工具调用结果分析  
第六步：加入日志  
第七步：加入失败处理  
第八步：加入评测样例  
第九步：扩展更多工具  
第十步：再考虑权限、审计、人工确认

如果是投放分析 Agent，第一版可以这样做：

只支持查询昨日数据  
只支持一个业务类型  
只支持整体指标和渠道拆分  
只输出分析报告  
不执行任何投放动作

等这个跑稳之后，再扩展到：

支持多个业务类型  
支持 adv_id / bundle / tagid 拆分  
支持作弊率分析  
支持日报自动发送  
支持生成操作建议清单  
支持人工确认后执行动作

这才是合理路径。

## 十三、第一版不要做什么

从 0 开始构建 Agent 时，最容易犯的错误是贪大。

第一版不要做这些：

不要做万能 Agent  
不要接入太多工具  
不要让 Agent 任意执行 SQL  
不要让 Agent 直接改生产数据  
不要让 Agent 自动发客户邮件  
不要让 Agent 自由访问所有系统  
不要一开始就做多 Agent  
不要一开始就做复杂长期记忆

第一版最重要的是跑通闭环：

用户提出任务  
Agent 理解任务  
Agent 调用工具  
Agent 拿到结果  
Agent 输出有依据的答案

只要这个闭环成立，就可以继续迭代。

## 十四、一个简单的 Agent 配置模板

你可以用下面这个模板设计任何 Agent。

Agent 名称：  
例如：投放诊断 Agent  
服务对象：  
例如：投放人员、运营人员、产品经理  
核心任务：  
例如：分析广告投放异常并输出优化建议  
输入：  
例如：业务类型、日期范围、指标、输出形式  
输出：  
例如：核心结论、关键数据、原因分析、建议动作  
可用工具：  
例如：`query_overview_metrics`、`query_channel_breakdown`、`query_fraud_metrics`  
不可执行动作：  
例如：不能自动暂停广告组、不能修改预算、不能发送客户邮件  
执行流程：  
例如：整体分析 -> 维度拆分 -> 异常定位 -> 原因归因 -> 建议输出  
失败处理：  
例如：数据为空则提示数据未就绪；参数缺失则追问用户  
评测标准：  
例如：是否命中异常原因、是否有数据支撑、是否输出可执行建议

这个模板非常适合从 0 到 1 搭建 Agent。

## 十五、以广告投放 Agent 为例的完整模板

Agent 名称：  
广告投放诊断 Agent  
服务对象：  
广告投放人员、运营人员、增长产品经理  
核心任务：  
根据投放数据诊断业务表现异常原因，并输出优化建议。  
输入：

- 业务类型：SOI / Nutra / HotSku / 杀毒 / 保险
- 时间范围：昨天 / 近 7 天 / 自定义日期
- 分析维度：channel / adv_id / bundle / tagid / country
- 输出形式：日报 / 异常分析 / 优化建议

输出：

1. 核心结论
2. 关键数据
3. 异常维度
4. 原因分析
5. 建议动作
6. 后续观察指标

可用工具：

- `query_overview_metrics`
- `query_channel_breakdown`
- `query_adv_breakdown`
- `query_bundle_breakdown`
- `query_fraud_metrics`
- `generate_report`

不可执行动作：

- 不自动暂停广告组
- 不自动修改预算
- 不自动调整出价
- 不自动发送客户邮件

执行流程：

1. 识别业务类型和时间范围
2. 查询整体指标
3. 对比历史基线
4. 拆分核心维度
5. 判断异常来源
6. 输出建议
7. 如涉及高风险动作，仅生成清单，不直接执行

失败处理：

- 参数缺失：追问用户
- 数据为空：提示数据未产出
- 工具失败：重试一次，失败后说明原因
- 结论不确定：标记为“需要进一步验证”

评测标准：

- 是否正确识别业务类型
- 是否查询核心指标
- 是否进行历史对比
- 是否定位异常维度
- 是否有数据支撑
- 是否输出可执行建议

## 十六、从 0 开始构建 Agent 的本质

构建 Agent 的本质，不是写一个很长的 Prompt。

而是设计一套小型任务执行系统。

这个系统至少包括：

任务边界  
输入输出  
Prompt  
工具  
Memory  
Controller  
Runtime  
日志  
失败处理  
评测

其中最核心的是：

明确任务  
控制流程  
调用工具  
基于真实结果输出

只要这四件事做好，一个基础 Agent 就能跑起来。

## 总结

从 0 开始构建一个 Agent，不要一开始追求复杂，也不要直接做万能系统。

正确路径是：

先选一个窄任务  
再定义输入输出  
然后设计 Prompt  
接着接入少量工具  
再加入 Memory 和日志  
最后加入重试、回退和评测

可以用一句话记住：

Agent 不是一个会聊天的模型，而是一个围绕目标持续执行、调用工具、观察反馈并修正行动的任务系统。

第一版 Agent 的目标不是完美，而是跑通最小闭环。

只要能做到：

理解任务 -> 调用工具 -> 观察结果 -> 输出结论

你就已经完成了从 0 到 1。
